# 학습 지침 개선 일지

> ALIBAI 프론트 학습 코치 운영을 하면서, 세션마다 드러난 마찰·요청·교훈을 기록한다.
> 여기 쌓인 패턴이 충분히 반복되면 `AGENTS.md`의 학습 지침으로 승격한다.

## 세션 1 (2026-06-07) — Day 1 타입 드릴 (`types/post.ts`)

작업: `Verdict` → `ExcuseContext` → `Post` → `Tag` → `NewPost` 정의.

### 학습자가 반복적으로 추가 요청한 것

- **큰 그림 먼저.** ("큰 그림과 오늘 작업범위를 모르는 상태", "이 파일이 뭔지부터")
  → 드릴 전에 앱 전체상 → 이번 주 범위 → 오늘 위치 → 이 파일 역할 순의 오리엔테이션이 필요.
- **용어/문법 뜻 먼저.** ("ts가 무슨 형식?", "verdict이 뭐냐구요", "export는 뭔데?", "interface 설명 빼먹었어")
  → 드릴에 등장하는 용어의 평이한 뜻과 문법 조각을, 쓰라고 하기 전에 깔아야 함.
- **기존 지식에 연결.** ("전역변수 선언?", "C언어의 헤더 파일이랑 비슷한건가?")
  → 아는 개념에 매핑하려는 성향. 비유를 적극 제공하고, 학습자가 꺼낸 비유는 정확히 교정.

### 코치(나)가 지적받은 점

1. **다짜고짜 드릴부터 냈다.** 가장 강한 피드백. 선행 문법 설명 없이 빈칸부터 채우라고 함.
2. **드릴에 쓴 키워드를 설명 안 함.** `interface`/`export`를 드릴에 등장시키고 개념은 누락.
3. **용어의 평이한 뜻을 건너뜀.** "Verdict이 뭐냐"에 타입 역할만 설명, 단어 뜻(=판결) 누락.
4. **기획서를 교조적으로 따름.** `situation`/`tags` 의미 중복을 내가 못 잡고 "계약대로 situation"
   을 권함. 학습자가 "그건 tag 아냐? 제목은 있어야지 게시판인데"로 더 정확히 정리.
   → 결과: `situation` 제거, `title` 추가, 분류는 `tags`. 기획서·드릴 문서도 함께 갱신.

### 매끄럽지 않았던 흐름

- 세션 1턴에 오리엔테이션 없이 곧장 `Verdict` 드릴 → 두 번 되돌려 큰 그림을 다시 깔았다.
- 개념→드릴 순서가 자주 역전됨 (드릴 먼저 던지고 개념 뒤따라 보충).
- 한 드릴의 선행 개념을 한 번에 다 깔지 않아 학습자가 "이것도 빼먹었다"고 채워 넣게 됨.

### 도출한 개선 방향 (지침 승격 후보)

1. **세션/파일 시작 시 [오리엔테이션] 단계 추가** — 첫 드릴 전에 "앱 전체 → 이번 주 범위 →
   오늘 위치 → 이 파일 역할". (현재 `[개념]→[드릴]→[체크]`는 너무 미시적.)
2. **[개념]은 드릴에 등장하는 모든 용어·문법·단어 뜻을 빠짐없이 선행.** "드릴에 쓸 거면 먼저 설명."
3. **드릴은 선행 개념이 다 깔린 뒤에만 낸다.** 개념↔드릴 순서 역전 금지.
4. **학습자가 아는 개념(C, 타 언어)에 적극 비유**하고, 학습자가 비유를 꺼내면 정확히 교정.
5. **기획서/스펙은 고정 계약이 아니라 학습자와 함께 다듬는 대상.** 학습자의 설계 판단이 타당하면
   코드+문서를 같이 갱신.

### 확정된 운영 규칙 (학습자 합의)

- **개념 주석은 요청받을 때만 작성한다.** 코치가 임의로 파일에 개념 요약 주석을 넣지 않는다.

## 세션 6 (2026-06-08) — Day 2 mock API (`api/posts.ts`)

작업: `fetchPosts` / `fetchPostById` / `createPost` 세 mock 함수 작성. `tsc -b`·`lint` 통과. 6단계 완료.
특징: 드릴 자체는 작았고 **개념 Q&A가 세션의 대부분**(비동기/Promise/배열메서드/타입문법). 전부 세션6 개념노트에 정리.

### 효과적이었던 것 (유지·강화할 패턴)

1. **2열 대조표.** 헷갈리는 쌍을 나란히: `매개변수 vs 인자`, `== vs ===`, `Promise<T>(타입) vs new Promise(값)`, `파괴적 vs 비파괴적`, `cursor vs nextCursor`. 학습자가 표 직후 바로 다음 질문으로 진행 → 인지 부하 감소 효과.
2. **빽빽한 한 줄을 토큰 단위로 분해.** 시그니처(`export async function ...`)·`await new Promise(...)`를 조각별 표로 쪼갬. "한 줄도 이해 안 된다"는 요청에 줄별·단어별 해설로 응답.
3. **함수 간 멘탈모델 재사용.** `filter` 비파괴성을 한 번 깔아두니 `concat` 버그를 같은 틀로 즉시 설명("filter 배울 때 그거"). 일관된 프레임이 복습 효과.
4. **설계 판단 즉응.** `tag→tagId` 제안 → 코드 수정+tsc 검증+설계 로그 기록을 한 사이클로 처리.

### 학습자가 준 피드백 (= 코치 미스)

1. **"파일 링크나 이름이라도 줘야지."** [오리엔테이션]은 깔았으나 **작업 대상 파일 경로/링크를 맨 앞에 명시 안 함.** (세션 1 "큰 그림 먼저" 교훈의 부분 회귀.)
2. **"fetchpost가 뭔지 안 알려줬어."** 함수 주변 동작(async 등)은 설명했지만 **그 함수가 무엇이고 왜 존재하는지**를 드릴 전에 안 짚음.
3. **"작성 전에 개념 설명이 부족했어."** 명시적. 드릴 대비 선행 개념이 얕음(특히 비동기).
4. **"그놈의 스위치. 비유 말고 확실한 표현으로."** `resolve`를 "스위치" 비유로만 반복 → **정확한 기술적 정의(상태 pending→fulfilled 전이)를 먼저 줬어야.**

### 매끄럽지 않았던 흐름

- `Promise<item: Post>` 오해가 **4턴 반복.** 처음엔 "봉투 불필요"(설계 논거)만 재진술 → 실제 막힘은 ①문법(`이름:타입`은 `{}` 안에서만, `<>` 바로 안엔 타입만) ②타입 검사기 동작(중괄호 쓰면 빨간 줄=반환값 불일치)이었다. 같은 질문 반복 = 핵심 미진단 신호.
- 비동기(동기/비동기·Promise·resolve·async 자동 포장) 질문이 연쇄 폭주 → 드릴 전 [개념]이 얇았다는 방증.

### 도출한 개선 방향 (지침 승격 후보)

1. **[오리엔테이션] 첫 줄에 `대상: <파일 경로+링크>` 고정.** 세션 1 "이 파일이 뭔지"를 **경로 명시**까지 강화.
2. **"개념 선행"을 용어 뜻 → "대상의 정체·존재 이유"까지 확장.** 드릴에 등장하는 함수/개념은 "이게 무엇이며 왜 필요한지"를 빈칸 제시 전에 한 문단 깔기.
3. **비유 의존도 조정: 정밀 정의 우선, 비유는 보조.** 학습자가 CS 배경이라 비유만 반복하면 역효과. 기본값을 "정확한 정의 먼저, 비유는 덤".
4. **같은 오해가 반복되면 원리 재진술 말고 막힌 층위를 바꿔 진단.** 설계/문법/타입검사기 동작을 분리해 어느 층에서 막혔는지 짚기.
5. **고밀도 주제(비동기 등)는 [개념]을 두껍게 선행.** 드릴 빈칸 1개당 개념 비중 2~3배. async/Promise는 드릴보다 개념이 본체.

## 세션 7 (2026-06-09) — Day 2 useEffect + 상세페이지 (`PostListPage.tsx`)

작업: `PostListPage`에서 `MOCK_POSTS` 직접 import 걷어내고 `useEffect`+`useState`로 `fetchPosts()` 연결, loading/error 3갈래. `tsc -b`·`lint` 통과(`posts={MOCK_POSTS}`→`posts={posts}`, `catch(e)`→`catch{` 2건 수정). PostListPage는 완료, PostDetailPage는 다음 세션.
특징: 드릴은 작았고, 세션 대부분이 "useEffect 콜백을 왜 async로 못 만드나(Promise 반환 문제)" 한 주제에 6~7턴 몰렸다.

### 학습자가 준 피드백 (= 코치 미스)

1. **외부 개념을 정의 없이 끌어다 썼다.** 어떤 걸 설명하려고 다른 개념(cleanup 함수, 동기 함수)을 도입하면서, 그 개념을 먼저 쉽게 설명하지 않고 곧장 등장시켰다. 학습자 말: "cleanup이 뭔지도 모르는데 갑자기 나와서 더 헷갈렸다." → **설명에 끌어오는 외부 개념도 드릴 용어처럼 먼저 짧고 쉽게 깔아야 한다.**
2. **방법을 먼저, 이유를 나중에 줬다.** 학습자가 직접 짚은 한 문장: "await를 쓰려는데 return type이 Promise가 되면 안 돼서 이 방식으로 했다." 이 인과 한 줄을 처음에 줬으면 바로 이해했다. 나는 우회 *방법*(load로 감싸기)을 먼저 설명하고 *이유*(await→async→Promise 반환→반환 슬롯과 충돌)는 뒤에 흩뿌렸다.
3. **같은 막힘을 6~7턴 반복하는 동안 규칙만 재진술했다.** `헬퍼로 넘긴다는 거야?` → `load가 뭘 돌려줘? 모순돼` → `그냥 밖에서 await하면?` → `인자로 넣고 반환 안하게 하면?` → `더 간단한 거 없어?` 같은 질문이 도는데, 막힌 전제(아래)를 바꿔 짚지 않고 결론(규칙)을 반복했다. (세션6 "층위 바꿔 진단"과 같은 실수의 재발.)

### 무엇이 결국 먹혔나

- 실제 막힌 전제는 "async를 끄면 되잖아"라는 가정이었다. `function a(){}`(→undefined) vs `async function b(){}`(→Promise<undefined>) **빈손 대조표**로 "async는 return을 안 써도 Promise를 반환한다(못 끈다)"를 보이자 풀렸다.
- 이어서 역할 분리표("바깥 콜백=반환 규칙 지키기 / 안쪽 load=await 쓰기")로 "왜 두 함수로 쪼개나"가 정착.
- 마지막에 실무 메뉴(IIFE / `.then` / 커스텀훅 / React Query)를 보이니 "복잡하다" 불만이 가셨다.

### 효과적이었던 것 (유지)

- **위치별 의미 대조표.** `{}`가 `=` 왼쪽이면 풀기·오른쪽이면 만들기 / import면 골라오기 / 함수·JSX면 별개 — "중괄호=풀어받기로 들었는데?" 혼란을 한 표로.
- **토큰 단위 분해표(B·C).** state 3줄·useEffect 블록을 기호별 표로. "C부터 같은 방식으로"를 학습자가 명시 요청 → 세션6 분해표 포맷이 계속 먹힘.
- **`< >`(타입) vs `( )`(값) 분리 + 추론 가능/불가 기준.** `useState(true)`는 `<>` 생략, `[]`·`null`은 명시.

### 대조표 샘플 (다음에도 같은 양식으로 재사용)

**① `{}` 위치별 의미** — "중괄호=풀어받기"로 뭉뚱그린 혼란을 가를 때.

| 어디에 나오나 | 하는 일 | 예시 |
|---|---|---|
| `=`의 왼쪽 | 풀어받기(구조분해) | `const {items} = data;` |
| `=`의 오른쪽 | 만들기(객체 리터럴) | `const data = {items: []};` |
| `import ___ from` 안 | 골라 가져오기(named import) | `import {useState} from "react";` |
| 함수/if/for 뒤 | 코드 블록 | `function f() { ... }` |
| JSX 안 | JS 값 꽂기 | `<p>{error}</p>` |

**② `useState` 한 줄 토큰 분해** — state 선언 문법 분해 시.

| 토큰 | 정체 | 역할 |
|---|---|---|
| `const` | 선언 키워드 | 재대입 금지(값은 setter로만 변경) |
| `[posts, setPosts]` | 배열 구조분해 | 0번=현재값 손잡이, 1번=변경 손잡이(순서 고정, 이름 자유) |
| `useState` | 훅 함수 | 칸 생성 + `[값, setter]` 반환 |
| `<Post[]>` | 제네릭 타입 인자 | TS 전용(빌드 후 소멸) |
| `([])` | 호출 인자=초기값 | 런타임 시작값(처음 한 번만) |

**③ 초기값별 `<>` 필요 여부** — "왜 어떤 건 타입 쓰고 어떤 건 안 쓰나".

| 초기값 | `<>` | 이유 |
|---|---|---|
| `true`·`0`·`"hi"` | 생략 | 추론됨(boolean 등) |
| `[]`(빈 배열) | 명시 | 원소 타입 모름(`never[]`로 빠짐) |
| `null` | 명시 | null만 보면 그 타입만 받음(`<string\|null>`) |

**④ 빈손 반환 대조 (async는 못 끈다)** — "async 끄면 되잖아" 가정을 깰 때.

| 함수 | return 썼나 | 실제 반환값 |
|---|---|---|
| `function a(){}` | 안 씀 | `undefined` |
| `async function b(){}` | 안 씀 | `Promise<undefined>` |

**⑤ try/catch/finally 실행 갈래** — 비동기 3갈래 처리 시.

| 경우 | try | catch | finally |
|---|---|---|---|
| 성공 | 실행(`setPosts`) | 건너뜀 | 실행(`setLoading(false)`) |
| 실패(throw) | 도중 중단 | 실행(`setError`) | 실행(`setLoading(false)`) |

### 도출한 개선 방향

1. **설명에 끌어오는 외부 개념은 먼저 짧고 쉽게 정의하고 쓴다.** 개념 선행은 드릴 용어뿐 아니라 코치 설명문에 등장하는 보조 개념(cleanup, 동기 함수 등)에도 적용.
2. **우회/패턴은 "왜 이렇게 하나"(인과 한 줄)를 방법보다 먼저.** "A를 하려는데 B 제약 때문에 C로 우회했다" 형태로 동기부터.
3. **같은 질문이 2~3회 돌면 규칙 재진술 멈추고, 막힌 전제(잘못된 가정)를 바꿔 짚는다.** 최소 대조 예시로 그 전제를 깨고 규칙에 연결.
4. **함수가 2개 얽히면 첫 등장부터 이름으로 분리**하고, 어느 함수의 반환을 누가 보는지 표로. 모호한 "콜백" 단독 사용 자제.
5. **"복잡하다/더 쉬운 거 없어?" 신호엔 방어 말고 인정 + 실무 메뉴부터.** 간단 변형(IIFE/`.then`)·실무 표준(React Query)을 보이고, "지금은 원리 체득용이라 raw로 간다"를 덧붙임.

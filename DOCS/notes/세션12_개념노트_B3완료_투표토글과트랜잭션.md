## 먼저 붙잡을 전체 그림

지난 세션(11)에서 posts·comments를 4계층(router→service→repository→schema/model)으로 깔고 동작시켰다. 이번 세션은 B3의 **마지막 조각이자 주인공인 votes(투표)**다.

```
B3 (CRUD + REST + 투표)
├─ schemas / models / repo / service / router  (posts)   ← 세션 11
├─ comments (평면 1:N)                                     ← 세션 11
└─ votes 토글 트랜잭션 + score ±   ★ 이번 세션 = 주인공
```

왜 votes가 "주인공"인가? posts·comments는 본질적으로 **"행 1개를 INSERT/SELECT"** 였다. 그런데 votes는 한 번의 요청이:

1. **먼저 읽고**(이 사람이 이미 투표했나?)
2. **판단하고**(같은 값? 반대 값? 처음?)
3. **여러 곳을 한꺼번에 바꾼다**(`votes` 행 + `posts.score`)

이 "여러 곳을 한꺼번에, 전부 되거나 전부 안 되게" 묶는 게 **트랜잭션**이고, 그게 이번 세션에서 배우는 단 하나의 큰 개념이다.

# Part 1. 트랜잭션(transaction) — 이번 세션의 주인공

## 1.1 정의 (용어부터)

- **transaction(트랜잭션, "거래·처리 단위")**: 여러 DB 변경을 **더 쪼갤 수 없는 하나의 단위**로 묶는 것. 결과는 둘 중 하나뿐 — "전부 성공" 또는 "전부 없던 일(rollback)". 중간 상태는 절대 남지 않는다.
- **atomic(어토믹, "원자적 = 더 못 쪼갬")**: 트랜잭션의 핵심 성질. 안에서 무슨 일이 있어도 바깥에는 "하기 전" 아니면 "다 한 후"만 보인다.
- **commit(커밋, "확정")**: 지금까지 쌓인 변경을 DB에 진짜로 영구히 박아 넣는 함수.
- **rollback(롤백, "되감기")**: 커밋 전까지의 변경을 전부 취소해 트랜잭션 시작 직전으로 되돌리는 함수.
- **ACID**: 트랜잭션이 지키는 4성질 머리글자 — Atomicity(원자성), Consistency(일관성), Isolation(격리성), Durability(영속성). 이번에 직접 쓰는 건 주로 **A(원자성)**.

## 1.2 등장 배경 (왜 생겼나 — 정의보다 먼저)

투표 한 번은 **두 곳**을 동시에 바꿔야 한다:

- `votes` 테이블에 표를 넣거나 / 지우거나 / 값을 바꾸고
- `posts.score`를 그만큼 ± 한다

트랜잭션이 없다고 상상해보자. `votes`에 표를 넣은 직후, `posts.score`를 갱신하기 **직전에** 서버가 죽으면? → **표는 있는데 점수는 안 오른** 영구적 모순이 DB에 남는다. 반대 순서면 점수만 오르고 표는 없다. 이런 "반쯤 된 상태"를 물리적으로 불가능하게 만들려고 트랜잭션이 나왔다. "둘 다 되거나, 둘 다 안 되거나."

## 1.3 실제 런타임 장면 (눈에 보이는 모양)

```python
session.add(vote)        # 또는 existing.value = ... / session.delete(existing)
post.score += value
```

이 두 줄을 실행해도 **이 순간 DB 파일은 안 바뀐다.** 변경은 세션(SQLAlchemy의 작업 공간) 안에 "대기 중"으로만 쌓여 있다. 이때 다른 커넥션에서 그 글을 조회하면 **옛날 점수**가 보인다.

그러다 `await session.commit()` **이 한 줄**에서 비로소 두 변경이 **동시에** DB에 반영되고, 그제야 다른 사람 눈에도 새 점수가 보인다. 만약 commit 전에 예외가 터지면 rollback되어 **둘 다 없던 일**이 된다 — score만 오르고 표는 사라지는 어중간한 상태가 메모리에서도 DB에서도 생길 수 없다.

## 1.4 flush vs commit — 다시 (세션 11에서 한 번, 여기서 또 막힌 부분)

이번에 처음 채울 때 마지막 줄에 `session.flush()`를 썼다가 코치가 `commit()`으로 고쳤다. 둘의 차이를 정확히:

| | flush | commit |
|---|---|---|
| 하는 일 | 메모리 변경을 DB로 **보냄**(SQL 실행) | 보낸 변경을 **확정**(영구화) |
| 확정? | ❌ 아직 트랜잭션 안, 취소 가능 | ✅ 영구. 이제 못 되돌림 |
| id 받기 | ✅ flush하면 자동 id 채워짐 | ✅ (commit도 내부적으로 flush 포함) |
| 세션 닫히면 | 확정 안 된 변경은 **rollback되어 사라짐** | 남아 있음 |

**왜 votes에서 flush로는 안 되나:** flush는 "보내되 확정 안 함"이라, 요청이 끝나 세션이 닫힐 때 commit이 없으면 변경이 **rollback되어 투표가 사라진다.** comments service에서도 마지막에 `commit`을 썼던 이유가 이것 — "이 작업을 진짜로 끝낸다"는 도장이 commit이다.

> 한 줄 요약(회수 질문 답): **flush는 트랜잭션 안에서 SQL만 미리 보내고 commit이 그 트랜잭션을 확정하므로, votes 행과 posts.score 두 변경을 영구히 남기려면 한 번의 commit으로 트랜잭션을 닫아야 한다.**

# Part 2. 멱등성·토글·UNIQUE — "조회→판단→갱신" 패턴이 생기는 이유

## 2.1 정의

- **idempotent(멱등, "여러 번 해도 결과 같음")**: 같은 요청을 두 번 보내도 상태가 한 번 보낸 것과 같아지는 성질.
- **toggle(토글, "켰다 껐다")**: 같은 버튼을 다시 누르면 이전 동작이 취소되는 동작.
- **UNIQUE 제약**: `votes(user_id, target_type, target_id)` 조합이 테이블에 **딱 1행만** 존재하도록 DB가 강제하는 규칙(`models.py`의 `UniqueConstraint`).

## 2.2 등장 배경

추천 버튼을 5번 연타하면 score가 +5가 되면 안 된다(부정 투표). "1인 1표"를 **앱 코드로만** 막으면 동시 요청 같은 데서 새는 구멍이 생긴다. 그래서 **DB의 UNIQUE 제약**으로 못을 박는다.

UNIQUE가 박히면 "그냥 INSERT 또 하기"가 불가능해진다(중복이면 DB가 거부). 그래서 어쩔 수 없이 **"이미 표가 있나 먼저 조회 → 상황 판단 → 갱신"** 이라는 패턴이 생긴다. 이게 votes service가 posts·comments보다 한 단계 복잡한 근본 이유다.

## 2.3 실제 런타임 장면 — 버튼 누르는 "영화"로 이해하기

표(table)로 세 줄을 따로 보면 잘 안 박힌다. **한 사람이 같은 글에서 버튼을 시간순으로 누르는 한 줄기 이야기**로 보면 자연스럽다.

무대: 글 1개(`id=10, score=0`), 내 표 없음. `나(user_id=1)`가 버튼을 누른다.

**🎬 1컷 — 👍(+1) 처음 누름** → 내 표가 *없으니* "처음" 갈래
```
[전] score=0, 내 표: 없음
  ↓ votes에 INSERT(value=+1), score += (+1)
[후] score=1, 내 표: +1     → 응답 {score:1, myVote:1}
```

**🎬 2컷 — 👍(+1) 또 누름** → 내 표(+1)와 들어온 값(+1)이 *같으니* "취소" 갈래
```
[전] score=1, 내 표: +1
  ↓ 내 votes 행 DELETE, score -= (+1)
[후] score=0, 내 표: 없음   → 응답 {score:0, myVote:0}
```
> 직관: 추천 버튼을 두 번 누르면 **추천 취소**(유튜브·레딧과 동일). 그래서 점수가 원래대로.

**🎬 3컷 — 👍 상태(score=1, 내 표=+1)에서 👎(-1)로 바꿈** → 내 표(+1)와 값(-1)이 *다르니* "전환" 갈래
```
[전] score=1, 내 표: +1
  ↓ 추천을 빼고(-1) 비추천을 더함(-1) = 합 -2
  ↓ 즉 score += (새값 - 옛값) = (-1) - (+1) = -2
  ↓ votes 행의 value를 +1 → -1로 수정(행은 1개 유지, UNIQUE)
[후] score=-1, 내 표: -1    → 응답 {score:-1, myVote:-1}
```

## 2.4 그래서 갈래는 딱 3개 (= service의 if/elif/else)

```
내 표 없음        → INSERT     → score += value           → myVote = value
내 표 == value    → DELETE     → score -= value           → myVote = 0
내 표 != value    → value 수정 → score += (value - 옛값)  → myVote = value
```

이게 멱등/토글의 구현이다. UNIQUE 덕분에 "내 표"는 항상 0개 아니면 1개라 분기가 깔끔하게 셋으로 떨어진다.

# Part 3. 투표 산수 — 내가 막혔던 지점들 (질문 모음)

## 3.1 질문: "실제 런타임 장면 표 이해 못했어"

→ 위 **2.3의 "영화"**가 그 답이다. 표의 세 줄은 따로 있는 규칙이 아니라, **한 사람이 버튼을 연달아 누를 때 거쳐가는 시간 순서**다. "지금 내 표가 어떤 상태냐"에 따라 다음 클릭이 어느 갈래로 가는지가 정해진다.

## 3.2 질문: "`(value - 0)` 여기 뭐지? / 싫어요→좋아요 변환도 가능한가?"

처음에 전환 갈래를 `post.score += (value - 0)`으로 채웠다. **`0`이 틀렸다.** 정답은 `value - existing.value`(옛값). 왜 깨지는지 싫어요→좋아요로 추적:

상황: 이미 👎(-1) 눌러둠(`score=-1`, `existing.value=-1`)에서 👍(+1) 누름(`value=+1`).
```
정답 (value - existing.value):  score += (+1) - (-1) = +2  →  -1 + 2 = +1  ✓
오답 (value - 0):               score += (+1) -  0   = +1  →  -1 + 1 =  0  ✗
```
추천 1표만 있으니 +1이 맞는데 `0`을 쓰면 0이 나온다. 이유: **옛날 -1표를 안 빼서.** 전환은 "옛 표 제거 + 새 표 추가" 두 가지가 동시에 일어나므로 `새값 - 옛값` 둘 다 들어가야 한다. `0`은 "옛 표 제거"를 빼먹은 식이다.

**싫어요→좋아요 변환, 된다.** 이 `else`(전환) 갈래는 방향을 안 가린다. 조건이 `existing.value != value`(값이 다르다)이기만 하면 들어오기 때문. `value - existing.value`가 **대칭식**이라 양방향이 자동으로 맞는다:
```
좋아요→싫어요:  (-1) - (+1) = -2
싫어요→좋아요:  (+1) - (-1) = +2
```
부호만 반대로 나올 뿐, 코드는 한 덩어리로 둘 다 처리한다.

## 3.3 질문: "value는 항상 1?"

**아니다. `value`는 +1 또는 -1**이다. 사용자가 👍를 누르면 body `{value:1}`, 👎면 `{value:-1}`로 들어와 router → service의 `value` 파라미터까지 그대로 흐른다. 항상 1이면 비추천 자체가 불가능하다.

헷갈리기 쉬운 두 변수를 반드시 구분:

| 이름 | 뜻 | 값 |
|---|---|---|
| `value` | **이번에 누른** 값(요청 body) | +1 또는 -1 |
| `existing.value` | DB에 **이미 저장된** 옛 표 | +1 또는 -1 |

전환 갈래(`else`)는 둘이 다를 때만 오므로 가능한 조합은 두 가지뿐: `(+1→-1)`, `(-1→+1)`.

**여기서 막판 버그가 나왔다.** "처음" 갈래를 `score += 1` / `my_vote = 1`로 채웠는데, 이러면 👎로 첫 투표해도 +1이 돼버린다. `value`(변수)를 써야 한다:
```python
# ① 처음
post.score += value     # (1을 박으면 비추천 망가짐)
my_vote = value
# ② 취소
post.score -= value     # (existing.value 써도 같음; 비추천 취소 시 +1 돼야 함)
my_vote = 0
```

# Part 4. votes 4계층 — 최종 코드 흐름

## 4.1 schemas.py (요청/응답 분리)

```python
class VoteIn(BaseModel):          # 요청: {value: 1 | -1}
    value: int
    model_config = ConfigDict(populate_by_name=True)

class VoteOut(BaseModel):         # 응답: {score, myVote}
    score: int
    my_vote: int = Field(alias="myVote")   # 내부 snake ↔ 외부 camel
    model_config = ConfigDict(populate_by_name=True)
```
- 요청과 응답을 **다른 스키마**로 나누는 이유는 posts에서와 동일(입력 필드와 출력 필드가 다름, 노출 통제).
- `my_vote ↔ myVote`는 alias. 응답을 camelCase로 내보내려면 라우터에서 `response_model_by_alias=True`.

## 4.2 repositories/votes.py (단순 조회만)

```python
async def get_vote(session, user_id, post_id) -> Vote | None:
    stmt = select(Vote).where(
        Vote.user_id == user_id,
        Vote.target_type == "post",   # ← 문자열! (다형 votes의 종류 구분)
        Vote.target_id == post_id,
    )
    return (await session.execute(stmt)).scalar_one_or_none()
```
- **막판 버그:** `Vote.target_type == Post`로 적었었다. `Post`는 (import도 안 된) 모델 클래스다. 여기 필요한 건 **문자열 `"post"`**.
- 왜 문자열이냐 → **다형(polymorphic) 테이블**이라서. `votes`는 글에도 댓글에도 달릴 수 있게 설계돼서, `target_type`("post"/"comment")+`target_id`로 "무엇에 대한 표인지"를 구분한다. 그래서 "이 글에 대한 내 표"는 `target_type="post"` AND `target_id=글id` AND `user_id=나`.
- `scalar_one_or_none()`: 결과가 0개면 `None`, 1개면 그 객체. UNIQUE라 2개 이상은 안 나온다. repo는 이렇게 **단순 조회**만, **판단(토글)은 service**.

## 4.3 services/votes.py (★ 주인공 — 조회→판단→갱신→commit)

```python
async def vote_post(session, post_id, user_id, value) -> tuple[int, int]:
    post = await get_post(session, post_id)
    if post is None:
        raise LookupError("post not found")      # 없는 글 → service는 신호만 던짐

    existing = await get_vote(session, user_id, post_id)

    if existing is None:                  # ① 처음
        session.add(Vote(user_id=user_id, target_type="post",
                         target_id=post_id, value=value))
        post.score += value
        my_vote = value
    elif existing.value == value:         # ② 같은 값 = 취소
        await session.delete(existing)
        post.score -= value
        my_vote = 0
    else:                                 # ③ 반대 값 = 전환
        post.score += (value - existing.value)
        existing.value = value
        my_vote = value

    await session.commit()                # votes 변경 + score를 한 트랜잭션으로 확정
    return post.score, my_vote
```
- **막판 버그:** `raise 404(...)`라고 적었었다. `404`는 정수라 호출 불가(TypeError). HTTP 상태코드는 **router의 책임**이고, service는 "글 없음"이라는 도메인 신호만 `LookupError`로 던진다.
- 두 테이블 변경(`votes` + `posts.score`)이 마지막 `commit` 하나로 **동시에** 확정 = 트랜잭션 경계.

## 4.4 routers/votes.py (HTTP만 번역)

```python
@router.post("/posts/{post_id}/vote", response_model=VoteOut, response_model_by_alias=True)
async def vote(post_id: int, body: VoteIn, session=Depends(get_db)):
    try:
        score, my_vote = await service.vote_post(
            session, post_id, user_id=1, value=body.value,   # user_id=1 임시 하드코딩
        )
    except LookupError as e:
        raise HTTPException(status_code=404, detail=str(e))   # 신호 → HTTP 번역
    return VoteOut(score=score, my_vote=my_vote)
```
- `user_id=1`은 임시 하드코딩(comments의 `author_id=1`과 동일) → B4 `get_current_user`로 교체.

## 4.5 main.py (라우터 등록)

posts·comments 옆에 import 한 줄 + `app.include_router(votes_router)` 한 줄. **이걸 빠뜨려서 `/docs`에 votes가 안 떴다** — "votes가 안 보인다"의 직접 원인은 거의 항상 등록 누락이다.

# Part 5. 에러 처리 — 내가 헷갈렸던 두 질문

## 5.1 질문: "except에 더 할 에러가 없는데?"

투표가 처리하는 에러는 **"없는 글에 투표"** 한 가지다. service가 `if post is None: raise LookupError(...)`로 던지고, router의 `except LookupError`가 그걸 받아 **HTTP 404**로 바꾼다. comments에서 이미 한 패턴 그대로다.

흐름: **service는 도메인 신호(LookupError)를 던지고 → router가 HTTP(404)로 번역한다.** 이렇게 나눠야 service가 HTTP를 몰라도 되고(다른 인터페이스에서 재사용 가능), router는 HTTP만 책임진다.

## 5.2 질문: "근데 똑같은 에러 하면 안 된다며?"

두 가지를 헷갈린 것. **정답을 그냥 베껴 쓰지 말라**(빈칸 드릴로 직접 채워라)는 것과, **코드 규칙을 일관되게 맞추는 것**은 다르다.

comments와 votes가 같은 `LookupError → 404`를 쓰는 건 베끼기가 아니라 **약속(convention)을 지키는 것**이다:
- service는 "글 없음"을 **항상 같은 신호**(LookupError)로 던진다 → 받는 쪽이 헷갈리지 않음.
- "글 없음"은 HTTP에서 **항상 404** → 프론트가 댓글이든 투표든 같은 코드로 같은 처리.

만약 엔드포인트마다 404/422/500 제각각이면 그게 진짜 잘못이다(프론트가 매번 다르게 대응해야 함). **규칙: 같은 의미의 에러는 어디서든 같은 코드로.** 단, 논리적으로 다른 에러는 당연히 다른 코드(잘못된 `value` 입력=422, 권한 없음=401).

## 5.3 질문: "같은 예외를 두 군데서 잡으면 겹쳐서 안 되는 거 아냐?"

안 겹친다. **try/except는 전역 등록이 아니라 "그 함수가 실행되는 동안, 그 try 블록 안에서 터진 것"만 잡는 지역(local) 장치**다.

```
요청 A: POST /posts/5/comments → create_comment() 의 try/except 만 작동
요청 B: POST /posts/5/vote     → vote() 의 try/except 만 작동 (완전 별개)
```
두 함수는 서로 다른 시점에 따로 호출된다. 같은 `LookupError` 타입을 잡아도 충돌·중복은 없다. (비유: 집집마다 현관에서 자기 택배만 받는다 — 옆집과 안 겹침.)

> 진짜 겹치면 안 되는 건 **라우트 경로**다. 같은 메서드+같은 경로를 두 번 등록하면 충돌. votes는 새 경로(`/posts/{id}/vote`)라 comments(`/posts/{id}/comments`)와 안 겹친다.

# Part 6. GET 응답에 votes 필드가 없는 이유 (질문)

## 6.1 질문: "post get에 votes 필드가 없네"

두 가지로 나뉜다.

**(1) 별도 `votes` 필드는 원래 없는 게 설계다.** 투표 정보는 **`score` 하나**로 표현한다 — `score = net = 추천수 − 비추천수`(집계된 정수 1개). 표 배열이나 추천/비추천 카운트를 따로 내려주지 않는다. 프론트는 "몇 점이냐"만 알면 되니까. 그래서 `GET /posts/{id}`의 `score`가 곧 votes 정보다. (`PostOut`에 `score: int` 있음.)

**(2) 진짜 빠진 건 `myVote`인데, 이건 B4 몫이다.** "내가 이 글에 추천/비추천했는지"(버튼 하이라이트용)는 **보는 사람이 누구냐에 따라 달라지는 값**이다(나는 +1, 옆사람은 0). 즉 **현재 로그인 사용자**를 알아야 계산된다. 지금은 `user_id=1` 하드코딩이고, 진짜 "현재 사용자"는 B4의 `get_current_user`에서 들어온다.

순서:
- **지금(B3):** `score`만 GET에 포함 → 정상. 투표 토글 자체(주인공)는 끝.
- **B4:** 인증 붙으면 `GET /posts`·`/posts/{id}`에 현재 사용자 기준 `myVote`를 votes **LEFT JOIN**(`user_id = 현재사용자`)으로 얹는다.

# Part 7. 동작 검증 (= B3 완료 판정)

`backend/.venv`로 서버 기동 후 `/docs`에서 같은 글에 순서대로:

```
1) value:1  → {score:1,  myVote:1}    처음 추천
2) value:1  → {score:0,  myVote:0}    재클릭 = 취소(원복)
3) value:1  → {score:1,  myVote:1}    다시 추천
4) value:-1 → {score:-1, myVote:-1}   전환(±2)
5) POST /posts/999/vote → 404          없는 글
```

전제: `users` id=1(`devuser`) seed + 글 1개 존재. 이 5개가 다 맞으면 **B3(CRUD + REST + 투표) 완료.**

# 한 장 요약

- **트랜잭션**: 여러 DB 변경을 "전부 되거나 전부 안 되거나"로 묶는 것. votes 행 + posts.score를 한 `commit`으로 동시 확정. `flush`는 보내기만(확정 X)이라 세션 닫히면 사라짐 → 반드시 `commit`.
- **토글 3갈래**: 표 없음→INSERT(`score += value`) / 같은 값→DELETE(`score -= value`, myVote=0) / 반대 값→value수정(`score += value - existing.value`, ±2). UNIQUE(1인1표)가 이 패턴을 만든다.
- **`value`(이번 클릭, ±1) ≠ `existing.value`(저장된 옛 표)**. 처음/취소 갈래에 `1`을 박으면 비추천이 깨진다 → 항상 `value` 변수.
- **`target_type="post"`**: 다형 votes에서 "무엇에 대한 표인지"를 문자열로 구분(모델 클래스 `Post` 아님).
- **에러 계약**: service는 `LookupError`(도메인 신호)만, router가 `404`(HTTP)로 번역. 같은 의미면 어디서든 같은 코드(일관성 = 베끼기 아님). try/except는 함수-지역이라 안 겹침.
- **GET에 votes 필드 없음**: `score`가 곧 투표 정보. 사용자별 `myVote`는 현재 사용자(B4 인증)가 있어야 LEFT JOIN으로 얹을 수 있어 B4로 미룸.
- **다음(B4)**: 해싱·JWT·`get_current_user`·CORS·rate limit + 프론트 axios 연결, 그리고 GET에 myVote 추가.

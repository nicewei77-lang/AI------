# 세션 11 개념 노트 — B3 라우터 계층(router) + 의존성 주입 + 실전 디버깅

> 대상 파일: `backend/app/routers/posts.py`(학습자 직접 작성), 보조로 `services/posts.py`(commit 추가)·`main.py`(라우터 등록).
> 진행분: **B3의 router 계층 완료(동작 검증).** posts에 대해 `GET /posts`(목록)·`GET /posts/{id}`(404)·`POST /posts`(201)·없는 태그(400)를 `/docs`로 전부 검증. 다음은 comments(평면) → votes(투표 토글 트랜잭션, B3의 주인공).
> 이 노트의 새 주제: **레이어드에서 router의 책임(HTTP만 안다) / APIRouter·include_router·prefix·tags / 의존성 주입(Depends·get_db) 깊게 / 트랜잭션 경계는 누가 소유하나(commit 위치 논쟁) / flush vs commit·expire_on_commit / Pydantic alias와 by_alias(왜 한 단어 필드는 똑같아 보이나) / 404 조건과 `is None` / HTTP 상태코드 매핑(201·400·404)과 ValueError→400 번역 / `/docs`(Swagger UI)가 뭔가 / import 경로와 패키지 루트 / 500의 진짜 원인은 터미널 트레이스백 / greenlet과 두 venv 함정 / FK 위반과 seed 유저 / BIGSERIAL 시퀀스는 왜 id가 3·15가 되나.**

## 먼저 붙잡을 전체 그림

지난 세션들에서 **HTTP를 모르는 부품들**을 먼저 만들었다:
- `schemas.py` — 데이터 모양(요청 `PostCreate` / 응답 `PostOut`)
- `models.py` — DB 테이블의 파이썬 얼굴
- `repositories/posts.py` — DB를 아는 함수(`get_post`·`list_posts`·`create_post`)
- `services/posts.py` — 규칙을 아는 함수(`create`: 태그 검증 후 INSERT)

이것들은 전부 **"문 없는 방"** 이었다. 바깥(브라우저·curl)에서 두드릴 입구가 없었다. 이번 세션이 그 **문**을 단다 — `router`. 라우터가 생겨야 비로소 HTTP 요청이 들어와 글이 만들어지고(201) 목록이 나온다.

### "CRUD + REST + 투표, 이거 아직 안 했는데?" (질문)
맞다. **B3 = `CRUD + REST + 투표`는 통째로 하나의 큰 단계**다. 그 안이 여러 조각으로 나뉜다:

```
B3 (CRUD + REST + 투표)
├─ schemas / models / repo / service.create   ← 이미 만든 "부품"(HTTP 없는 조각)
├─ router  ← 이번 세션 (= CRUD를 REST로 노출하는 입구)
├─ comments (평면 1:N)
└─ votes 토글 트랜잭션 + score ±    ← "투표", 맨 마지막 주인공
```

즉 "CRUD를 REST로 동작시키는 입구"가 router다. 부품을 아무리 잘 만들어도 router가 없으면 "동작"이 0이다. **백엔드는 동작이 곧 진도**라, 이 문을 달고 curl/`/docs`로 201·404·400을 눈으로 확인해야 진짜 완료다.

# Part 1. 레이어드 아키텍처에서 router의 책임

## 1.1 router는 "HTTP만 아는" 계층

레이어드 구조는 한 요청을 네 계층으로 쪼갠다:

```
요청 → router(입출력) → service(규칙) → repository(DB) → model/schema(데이터 모양)
```

각 계층의 판별 질문은 **"이 코드가 무엇을 아는가"**:
- **HTTP를 아는가** → router (상태코드·요청 파싱·응답 JSON 모양)
- **규칙/유스케이스를 아는가** → service (태그 검증, 트랜잭션 조립)
- **DB를 아는가** → repository (SELECT/INSERT 쿼리)
- **데이터 모양인가** → schema/model

router의 책임은 딱 **HTTP 번역**이다. "이건 201이다 / 404다 / 400이다", "요청 body를 `PostCreate`로 받는다", "응답을 camelCase JSON으로 낸다" — 이게 전부다. **`Post` 객체가 어디서 왔는지, SQL이 어떻게 나갔는지는 모른 채** 입출력만 책임진다.

## 1.2 이번에 단 세 개의 문

```python
@router.get("/posts")             # 목록  → repo.list_posts
@router.get("/posts/{post_id}")   # 상세  → repo.get_post, 없으면 404
@router.post("/posts")            # 작성  → service.create, 성공 201
```

- 목록·상세는 단순 조회라 router가 **repository를 직접** 부른다.
- 작성은 "태그 검증 + 트랜잭션"이라는 규칙이 있어 **service를 거친다**.
  (규칙이 있으면 service, 단순 조회면 repo 직접 — 절대 법칙은 아니고 책임 기준이다.)

# Part 2. APIRouter — 라우터를 파일로 쪼개는 도구

## 2.1 정의·등장 배경

- **정의:** `APIRouter`("API 경로 모음")는 **엔드포인트들을 한 묶음으로 모아 두는 작은 라우터 객체**다. `app`(FastAPI 본체)에 직접 `@app.get`을 달지 않고, `router = APIRouter()`에 경로를 달아 두었다가 나중에 `app.include_router(router)`로 **본체에 끼워 넣는다**(include=포함시키다).
- **등장 배경:** 처음엔 다들 `main.py` 하나에 `@app.get(...)`을 전부 붙인다. 그런데 posts·comments·votes·auth가 늘면 `main.py`가 수백 줄로 비대해진다(= Flask 뷰 비대화의 재현). "기능별로 파일을 쪼개되, 본체엔 한 줄로 합치고 싶다" → 그래서 `APIRouter`가 나왔다. 우리가 `routers/` **폴더**를 둔 이유와 정확히 같다.

## 2.2 실제 런타임 장면 — 등록해야 문이 열린다

`router`만 있을 땐 어떤 URL도 응답하지 않는다. `main.py`에서 `app.include_router(router)`를 부르는 **그 순간** `/posts`가 라우팅 테이블에 등록되고, 그제야 요청이 200을 받는다. **등록을 깜빡하면 전부 404** — 흔한 함정이다.

```python
# main.py
from fastapi import FastAPI
from app.routers.posts import router    # ← import 경로 주의(Part 8)

app = FastAPI()
app.include_router(router)              # ← 이 줄이 문을 앱에 단다
```

## 2.3 prefix와 tags (질문: 문서 직접 읽고 옴)

문서 정의:
- **`prefix`** (str, 기본 `''`) — 이 라우터의 모든 경로 앞에 자동으로 붙는 **공통 머리말**.
- **`tags`** (list[str], 기본 None) — 이 라우터 경로들에 붙는 **분류 태그**. OpenAPI(`/docs`)에 반영된다.

우리 라우터에 적용하면:

```python
router = APIRouter(prefix="/posts", tags=["posts"])

@router.get("")              # 실제 경로 = /posts      (목록)
@router.get("/{post_id}")    # 실제 경로 = /posts/{id} (상세)
@router.post("", status_code=201)   # 실제 경로 = /posts (작성)
```

- **prefix 함정:** prefix는 **앞에 덧붙이는** 거라, prefix를 `"/posts"`로 줬는데 데코레이터에도 `"/posts"`를 또 쓰면 경로가 `/posts/posts`가 되어 404가 난다. prefix를 쓰면 데코레이터엔 **나머지만**(`""`, `"/{post_id}"`) 적는다.
- prefix 끝에 슬래시(`/`)를 붙이지 않는다(`"/posts/"` ❌). FastAPI 규칙: "prefix는 `/`로 시작하고 `/`로 안 끝남."
- **`tags=["posts"]`** — 기능엔 영향 없고, **`/docs`에서 엔드포인트를 "posts" 그룹으로 묶어** 보여줄 뿐이다. comments·votes 라우터가 생기면 각자 `tags`로 /docs가 기능별로 깔끔히 갈린다. (이번엔 tags를 안 줘서 전부 `default` 그룹에 묶였는데, 무해하다.)

# Part 3. 의존성 주입(Dependency Injection) — Depends·get_db

> 이 세션에서 가장 깊게 판 주제. "의존성이 정확히 뭔지 몰랐다"는 질문에서 출발했다.

## 3.1 의존성이란 (정의)

- **의존성(dependency)** = 어떤 코드가 돌기 위해 **먼저 준비돼 있어야 하는 다른 값/객체**. (depend on = ~에 기대다.) 우리 핸들러엔 **DB 세션**이 의존성이다 — 세션 없이는 쿼리를 못 날리니까.
- **주입(injection)** = 그 의존성을 함수가 **직접 만들지 않고, 바깥이 만들어서 인자로 "넣어준다"**.
- 합치면 **의존성 주입** = "필요한 준비물을 내가 만들지 말고, 누가 만들어 인자로 꽂아달라"고 선언하는 방식.

처음 가졌던 이해("한 모듈이 다른 모듈의 결과를 받아야 작동하는 방식")는 **절반 맞다**. 빠진 핵심은 *"그 결과를 누가 만들어 넣어주느냐"* — 답은 **바깥(FastAPI 프레임워크)** 이다.

## 3.2 Depends의 핵심 한 줄

```python
from app.db import get_db

@router.get("/posts")
async def list_posts(session: AsyncSession = Depends(get_db)):
    ...
```

- `Depends(get_db)` = "이 인자(`session`)는 내가 채우지 마. **`get_db`를 먼저 실행해서 그 반환을 여기 넣어줘**"라는 선언.
- 문서의 핵심 문장: **"Don't call it directly, FastAPI will call it for you."**
  → 내가 `get_db()`를 호출하는 게 아니라 **이름만**(`Depends(get_db)`, 괄호 없이) 넘기면 FastAPI가 호출 시점을 정해 대신 부른다. 이 뒤집힘을 **제어의 역전(Inversion of Control)** 이라 부른다.
- **함정:** `Depends(get_db())`처럼 **괄호를 붙이면 틀림.** 그건 지금 당장 한 번 실행해 결과를 박아버리는 거라, 매 요청 새 세션을 못 받는다. 반드시 **함수 이름만** 넘긴다.

## 3.3 왜 생겼나 (등장 배경)

DI 없던 시절엔 핸들러 안에서 직접 준비했다:

```python
async def get_post(post_id):
    session = SessionLocal()          # 내가 직접 만들고
    try: ...
    finally: await session.close()    # 내가 직접 닫고
```

문제 둘:
1. **중복** — 이 세션 만들고 닫는 코드가 모든 핸들러에 복붙된다.
2. **테스트 불가** — 진짜 DB 세션이 코드에 박혀 있어 가짜로 바꿔치기할 수가 없다.

→ "준비물 만드는 책임을 한 함수(`get_db`)로 빼고, 핸들러는 **완성품만 받자**" → DI가 나왔다.

## 3.4 실제 런타임 장면

요청이 들어오면 FastAPI가 핸들러 **시그니처를 스캔** → `session: AsyncSession = Depends(get_db)`를 보고 *"이 인자는 get_db를 돌려 채워야겠군"* → `get_db()` 실행 → `yield session`의 그 session을 **인자에 꽂음** → 핸들러 본문 실행 → 끝나면 `yield` **다음 줄**(세션 닫기)이 돌아간다.

```python
# db.py
async def get_db():
    async with SessionLocal() as session:
        yield session          # ← 여기서 멈춰 session을 핸들러에 빌려줌
        # 핸들러 끝나면 async with 탈출 = 세션 자동 반납/닫힘
```

즉 **요청 1개 = 세션 1개**가 자동으로 열리고 닫힌다. 내 눈엔 핸들러가 `session`을 **이미 받은 채로** 시작하는 걸로 보인다 — 만드는 과정이 안 보이는 게 DI의 효과다.

## 3.5 "보통 의존성을 만드는 건 어떤 상황? 효율적인 코드 만들 때?" (질문)

핵심 교정: **효율(성능)이 주목적이 아니다.** 진짜 신호는 *"여러 핸들러가 같은 준비/검증을 반복하는데, 그게 핸들러의 본업이 아닐 때"*. 본업이 아닌 잡일을 바깥으로 빼는 것이다. 대표 4상황:

1. **요청마다 열고 닫아야 하는 자원(lifecycle)** — DB 세션, 파일 핸들, 외부 API 클라이언트. → `get_db`가 이것.
2. **여러 엔드포인트가 공유하는 선행 조건/검증** — "로그인돼 있어야 함". → B4의 `get_current_user`: 토큰을 까서 유저를 주거나 없으면 401을 대신 던진다. `POST /posts`·`POST /comments`·`POST /vote` 전부 같은 검증이 필요한데 복붙 대신 의존성 하나로.
3. **반복되는 공통 파라미터 묶음** — 문서 예시 `common_parameters(q, skip, limit)`처럼 페이징/검색 파라미터를 여러 목록 엔드포인트가 공유할 때.
4. **설정·전역 자원 주입** — `settings`, Redis 클라이언트 등(지금 단계엔 안 씀).

그래서 "효율"이라는 표현은 다음 3개로 바꿔 이해해야 정확하다:
- **재사용** — 같은 준비 코드를 N개 핸들러에 복붙 안 함.
- **책임 분리** — 핸들러는 "무엇을 할지"만, 준비물은 의존성이.
- **교체 가능성** — 테스트 때 `get_db`를 가짜 세션으로 바꿔치기(`dependency_overrides`). 진짜 DB 없이 테스트.

> 성능 얘기가 전혀 없진 않다: `Depends`엔 `use_cache=True`가 있어 **한 요청 안에서 같은 의존성을 여러 번 부르면 처음 결과를 재사용**한다. 하지만 이건 덤이지 목적이 아니다.

**판단 기준 한 줄:** "이 잡일이 핸들러의 본업인가? 아니고, 다른 핸들러도 똑같이 필요로 하나? → 그럼 의존성으로 뺀다."

## 3.6 문법 두 가지 스타일

```python
# 스타일 A (이번에 쓴 것)
session: AsyncSession = Depends(get_db)
# 스타일 B (문서 권장, 최신)
commons: Annotated[dict, Depends(common_parameters)]
```

둘 다 맞다. `Annotated`가 최신 권장형. 지금은 A로 가도 충분 — 통일만 하면 된다.

# Part 4. 트랜잭션 경계는 누가 소유하나 — commit 위치 논쟁

> "라우터는 받기만 하는 거 아냐? 여기서 commit 해서 뭐해?" (질문) — 아주 정확한 직감이었다.

## 4.1 commit은 router의 일이 아니다

router의 책임은 **HTTP 번역**뿐이다(201/404/400). 그런데 `commit`은 HTTP 개념이 아니라 **"이 유스케이스(글 작성)가 성공했으니 DB에 확정한다"** 는 **트랜잭션 경계** 개념이다.

- **트랜잭션 경계** = "어디부터 어디까지가 한 덩어리로 성공/실패해야 하나".
- 이걸 아는 건 그 작업을 조립한 **service**지 문지기(router)가 아니다.

→ 그래서 commit은 router에서 빼고 **service.create 끝**에 둔다.

## 4.2 flush vs commit — 왜 commit이 꼭 필요한가 (숨은 함정)

repository의 `create_post`는 `session.flush()`까지만 한다.

- **flush** = 세션에 쌓인 SQL을 DB로 **보내서 id를 받아오되, 아직 확정 전**(트랜잭션 안의 임시 상태).
- **commit** = 그 변경을 **DB에 영구 반영(확정)**.

`get_db`도 commit을 하지 않는다. 그러면? 요청이 끝나 `async with`가 세션을 닫을 때 **자동 롤백** → INSERT가 사라진다. **POST가 201을 돌려줬는데 `GET /posts`엔 안 보이는 "유령 생성"** 이 된다. 그래서 누군가 commit을 해야 하고, "성공했으니 확정"의 판단은 유스케이스를 조립한 service의 책임이다.

```python
# services/posts.py
async def create(session, body, author_id):
    if len(tags) != len(set(body.tag_ids)):
        raise ValueError(...)            # ① 검증 실패 → 여기서 탈출(commit 도달 X)
    post = await create_post(session, ...)  # ② INSERT(flush)
    await session.commit()               # ③ 모든 쓰기 후, return 직전 확정
    return post
```

## 4.3 commit 위치가 왜 그 자리인가 (질문)

```
검증(raise) 다음 → 모든 INSERT 다음 → return 직전
```

- **검증 다음:** ValueError로 빠지면 commit 줄에 도달조차 안 한다 → 잘못된 입력은 절대 확정 안 됨. "성공해야만 확정"이 흐름으로 지켜진다.
- **모든 INSERT 다음:** 지금은 post 하나뿐이지만, 곧 투표에서 *여러 행을 바꾸고 마지막에 한 번 commit* 한다. "이 유스케이스의 모든 쓰기가 끝난 지점"에 두는 습관이 그대로 이어진다.
- **return 직전:** commit이 끝나야 진짜 저장된 post를 돌려주는 거라 순서상 자연스럽다.

## 4.4 expire_on_commit=False 가 여기서 빛난다

보통 commit하면 ORM 객체 속성이 **만료(expire)** 되어, `return post` 뒤 `PostOut`이 `post.title`을 읽을 때 DB를 다시 쳐야 한다(만료된 비동기 객체라 에러 나기 쉬움). [db.py]에서 세션을 `expire_on_commit=False`로 만들어 둔 덕에 **commit 후에도 속성이 그대로 읽힌다.** 그래서 commit **후 return**해도 안전하다. 과거에 깐 설정이 이 세션에서 보상으로 돌아온 장면.

> commit을 service에 두는 또 다른 선택지: `get_db`에서 요청 성공 시 자동 commit. 둘 다 맞지만, **투표(여러 행 한 트랜잭션)** 가 곧 주인공이라 "유스케이스를 조립하는 service가 트랜잭션을 소유"하는 그림을 미리 박아두는 게 일관적이라 service를 택했다.

# Part 5. Pydantic alias와 by_alias — camelCase 출력

## 5.1 alias가 하는 일

```python
class PostOut(BaseModel):
    excuse_text: str = Field(alias="excuseText")   # 내부 snake ↔ 외부 camel
    created_at: datetime = Field(alias="createdAt")
```

- **alias(별칭)** = 같은 필드의 **외부 이름**. 내부 파이썬 필드는 `excuse_text`(snake_case), 외부 JSON 키는 `excuseText`(camelCase). 프론트가 본문만 axios로 바꿔 붙으려면 JSON 키가 camel이어야 한다(프론트 `types/post.ts`가 `post.excuseText`로 읽음).

## 5.2 by_alias는 "어떤 alias를 쓸지"가 아니라 "alias를 쓸지 말지" 스위치

```python
.model_dump(by_alias=True)
response_model_by_alias=True
```

- `by_alias`의 **타입은 boolean**(True/False). **어떤 alias를 쓸지는 이미 `Field(alias=...)`에 박혀 있고**, `by_alias`는 그걸 **출력에 적용할지 말지**만 정한다.
- `by_alias=False` → `{"excuse_text": ...}` (필드명, snake)
- `by_alias=True` → `{"excuseText": ...}` (alias, camel)

### "둘 다 카멜케이스여도 스네이크랑 똑같은데?" (질문)
정확한 관찰이다. **한 단어짜리 필드는 camel이든 snake든 똑같다** — `title`·`id`·`score`엔 밑줄이 없으니 차이가 없다. `by_alias`가 **눈에 보이게 갈리는 건 밑줄 있는 필드뿐**:

```jsonc
// by_alias=False
{ "id":1, "title":"지각", "excuse_text":"...", "created_at":"...", "score":0 }
// by_alias=True
{ "id":1, "title":"지각", "excuseText":"...", "createdAt":"...", "score":0 }
//                        ^^^^^^^^^^         ^^^^^^^^^   이 둘만 바뀜
```

**왜 이 두 개가 치명적이냐:** 프론트가 `post.excuseText`로 읽는데 `by_alias=False`로 내보내면 키가 `excuse_text`라 `post.excuseText === undefined` → 화면에 변명 본문이 안 뜬다. 한 단어 필드가 우연히 같아 보여 "차이 없네" 싶지만, **계약이 깨지는 지점은 정확히 밑줄 필드**다. → 그래서 둘 다 `True`.

## 5.3 두 출력 방식

- 목록(dict 반환): 직접 `PostOut.model_validate(p).model_dump(by_alias=True)`
- 단일(response_model 사용): 데코레이터에 `response_model=PostOut, response_model_by_alias=True`

후자는 FastAPI가 ORM 객체(`post`)를 `PostOut`으로 자동 변환·직렬화해 준다(`from_attributes=True` 덕).

# Part 6. HTTP 상태코드 매핑과 에러 번역

## 6.1 상태코드는 프로토콜 레벨의 "결과 분류"

- **201 Created** — 작성 성공. `@router.post(..., status_code=201)` — 데코레이터의 status_code는 **"성공했을 때 줄 코드"**.
- **404 Not Found** — 자원 없음.
- **400 Bad Request** — 잘못된 입력.
- **422 Unprocessable Entity** — Pydantic 검증 실패(타입·필수 누락). FastAPI가 자동으로 냄.

함수가 그냥 `return None`이나 `return {"error":...}`을 하면 상태코드는 여전히 200이라 프론트가 "성공했는데 본문이 이상하네?"로 오해한다. **없으면 진짜 404, 틀리면 진짜 400**을 프로토콜로 말해야 프론트가 `if (res.status===404)`로 분기한다.

### 검토에서 잡은 버그: status_code=400을 데코레이터에 둠
작성 핸들러 데코레이터에 `status_code=400`을 두면 **정상 작성이어도 응답이 400**으로 나가 프론트가 실패로 오해한다. 데코레이터 = **성공 코드(201)**, 400은 **실패 경로(except 안)에만**. 둘을 제자리에 두는 게 REST 상태코드 설계의 핵심.

## 6.2 404 조건과 `is None`

```python
post = await repo.get_post(session, post_id)
if post is None:
    raise HTTPException(status_code=404, detail="post not found")
return post
```

- repo의 `get_post`는 `scalar_one_or_none()` → "있으면 그 행, **없으면 None**". 그래서 "없을 때 값"은 `None`이고, 그걸 검사한다.
- **`== None`이 아니라 `is None`:** `None`은 프로그램에 **딱 하나뿐인 객체**라, "값이 같나(`==`)"가 아니라 "**그 객체 자체인가(`is`)**"로 검사하는 게 정확하고 빠르다. `==`은 클래스가 `__eq__`로 뜻을 바꿀 수 있어 함정 여지가 있다. (린트 E711 경고도 `== None`을 잡는다.)

## 6.3 HTTPException + ValueError → 400 번역 (복선 회수)

```python
@router.post("/posts", status_code=201, response_model=PostOut, response_model_by_alias=True)
async def create_post(body: PostCreate, session: AsyncSession = Depends(get_db)):
    try:
        post = await service.create(session, body, author_id=1)
    except ValueError as e:
        raise HTTPException(status_code=400, detail=str(e))
    return post
```

- `HTTPException` = "이 요청을 여기서 중단하고 이 상태코드로 응답해"라고 던지는 예외.
- **복선:** `service.create`는 없는 태그 slug가 섞이면 `raise ValueError(...)`를 던진다. **service는 일부러 HTTP를 모른다.** router가 안 잡으면 예외가 그대로 위로 터져 **500 Internal Server Error**가 난다("내 서버 버그"처럼 보이지만 실은 "네 입력이 틀림"). 그래서 router가 `except ValueError → HTTPException(400)`으로 **번역**한다.
- 검증으로 확인: 없는 태그(`tagIds:["없는태그"]`)로 POST → **500이 아니라 400**이 떴다. 번역 성공.

# Part 7. /docs (Swagger UI) — 자동 생성 문서 + 시험장

> "docs가 뭐야?" (질문)

- `http://localhost:8000/docs` — FastAPI가 **자동 생성**하는 **Swagger UI** 화면.
- **정의:** 내가 만든 엔드포인트가 목록으로 뜨고, 각각 어떤 입력(`PostCreate`)을 받고 어떤 출력(`PostOut`)을 주는지 표로 정리된다.
- **왜 있나:** 보통 API 문서는 손으로 적어 코드와 금방 틀어진다. FastAPI는 **내가 쓴 타입(Pydantic 스키마)·데코레이터를 읽어 문서를 자동 생성**해 절대 안 틀어진다(= "타입 기반"의 보상).
- **실제 장면:** 단순 문서가 아니라 **눌러서 실행**할 수 있다. 엔드포인트 펼치고 → `Try it out` → 입력 → `Execute` → curl 없이 브라우저에서 요청이 나가고 응답·상태코드가 화면에 뜬다.
- `/redoc`(같은 내용 다른 디자인)도 자동으로 생긴다.

# Part 8. import 경로와 패키지 루트

### 검토에서 잡은 버그: `from routers.posts import router`
```python
from routers.posts import router        # ❌ ModuleNotFoundError: No module named 'routers'
```

이 프로젝트는 `uvicorn app.main:app`으로 띄운다 → 패키지 최상단이 **`app`** 이라, 모든 import가 `from app.xxx`로 시작해야 한다. `routers`만 단독으로는 sys.path에 없다. 에디터에선 `routers/`가 바로 옆 폴더라 될 것 같지만, 실행은 `app` **패키지 바깥(backend/)** 에서 시작된다. **import는 "에디터 위치"가 아니라 "실행 시 패키지 루트" 기준.** (그래서 `uvicorn`도 `cd backend` 후 실행해 `app`을 찾게 한다.)

# Part 9. 실전 디버깅 — 동작 검증에서 만난 4개의 벽

> 백엔드는 "돌려보면 안다." 서버가 떠도 엔드포인트를 실제로 두드려야 진짜 검증이다. POST 한 번에 벽이 연달아 나왔다.

## 9.1 500의 진짜 원인은 브라우저가 아니라 터미널

`/docs`에서 POST → **500 Internal Server Error**, 본문엔 그냥 `Internal Server Error`. 이 메시지는 **일부러 뭉뚱그린 것**(서버 내부를 외부에 안 흘리려고). **진짜 원인은 `uvicorn` 터미널의 트레이스백**에 찍힌다. 디버깅 1순위 = 서버 로그의 마지막 몇 줄.

## 9.2 벽 ①: greenlet 없음 + 두 venv 함정

```
ValueError: the greenlet library is required to use this function. No module named 'greenlet'
```

- **greenlet** = SQLAlchemy 내부의 동기 코드를 `async/await`에 끼워 돌리기 위한 **동기↔비동기 다리** 라이브러리. 비동기 DB 작업을 하나라도 하면 무조건 필요하다.
- **원인:** 프로젝트에 venv가 **두 개**였다 — 루트 `.venv`와 `backend/.venv`. 루트 `.venv`엔 greenlet이 빠져 있었고, 그걸 켠 채로 서버를 띄웠다. `backend/.venv`엔 requirements가 다 깔려 있다.
- **해결:** `backend/.venv`로 전환.
  ```bash
  deactivate
  cd backend && source .venv/bin/activate
  which python      # → backend/.venv/bin/python 확인
  uvicorn app.main:app --reload
  ```
- **교훈:** 이 프로젝트는 **항상 backend/.venv로 실행.** (루트 .venv는 greenlet 없음.)

## 9.3 벽 ②: FK 위반 — author_id=1 유저가 없음

```
ForeignKeyViolationError: insert or update on table "posts" violates
foreign key constraint "posts_author_id_fkey"
DETAIL:  Key (author_id)=(1) is not present in table "users".
```

- **외래키(FK)** = `posts.author_id`는 `users(id)`를 가리키는 제약. "글에는 반드시 존재하는 작성자가 있어야 한다." 로그인(B4)이 아직 없어 임시로 `author_id=1`을 박았는데, `users`에 id=1 유저가 없어 DB가 INSERT를 **거부**했다.
- 제약은 **commit(확정) 순간 최종 검사**된다("비번 틀려도 통과하다 첫 쿼리서 터짐"과 같은 결 — flush는 통과해도 commit에서 운다).
- **해결:** seed 유저 한 명 삽입.
  ```bash
  docker exec alibai-db psql -U alibai -d alibai -c \
    "INSERT INTO users (id, username, email, password_hash) VALUES (1, 'devuser', 'dev@example.com', 'dev-placeholder');"
  ```
  (`password_hash`는 인증 전이라 임시 문자열. B4에서 bcrypt 해시로 교체.)

## 9.4 벽 ③: BIGSERIAL 시퀀스 — 왜 id가 3, 15가 됐나

처음 `INSERT INTO users (...)`(id 미지정)를 하니 `INSERT 0 1`은 떴는데 여전히 FK 에러. 확인해 보니 유저 **id가 1이 아니라 3**이었다.

- **BIGSERIAL** = id를 **시퀀스(자동 증가 카운터)** 에서 뽑아 쓴다. 이 카운터는 **한 번 뽑으면 되돌아가지 않는다** — INSERT가 실패하거나 여러 번 시도하면 1·2가 소진되고 3까지 올라간다. 즉 **시퀀스는 "행 개수"가 아니라 "누적 발급 수"** 를 따라간다. (그래서 "테이블 비었으니 무조건 1"은 틀린 가정.)
- 같은 이유로 posts의 첫 작성 글 id가 **15**로 찍혔다(앞선 실패 시도들이 시퀀스를 올림). 비어 있어도 id가 1부터가 아닐 수 있다 — 정상이다.
- **해결:** `author_id=1`을 의미 있게 유지하려고 **id=1을 명시 지정**해 유저를 넣었다(시퀀스를 거치지 않음).

## 9.5 최종 통과 — 201 + 계약 검증

```json
// POST /posts → 201 Created
{ "id": 15, "title": "지각", "excuseText": "차가 막혀서",
  "createdAt": "2026-06-13T08:36:34.920270Z", "score": 0,
  "verdict": null, "credibility": null, "context": null, "tags": [] }
```

한 번에 검증된 것:
- ✅ **201** — 작성 성공 상태코드
- ✅ **camelCase** — `excuseText`·`createdAt`(snake 아님) → alias 계약 OK
- ✅ **score: 0** — 기본값(투표 전)
- ✅ 로그에 **COMMIT** 찍힘 → service의 commit 동작 = DB에 진짜 저장
- ✅ `GET /posts` 목록에 보임(왕복) / `GET /posts/99999` → 404 / 없는 태그 POST → 400

# 한 줄 요약 (복습용 체크)

1. **router = HTTP만 아는 계층.** 상태코드·요청파싱·응답JSON 모양만 책임, DB·SQL은 모름.
2. **APIRouter**로 기능별 파일 쪼개고 `app.include_router`로 본체에 합침(안 하면 404). prefix는 공통 머리말, tags는 /docs 그룹.
3. **의존성 주입** = 준비물을 핸들러가 안 만들고 바깥(FastAPI)이 만들어 인자로 꽂음. `Depends(get_db)`(괄호 없이 이름만). 목적은 효율 아니라 **재사용·책임분리·교체가능성**.
4. **commit은 router가 아니라 service(트랜잭션 경계 소유).** flush=임시 전송, commit=확정. 안 하면 유령 생성.
5. **by_alias=True** = alias(camel) 출력 스위치. 밑줄 필드만 눈에 띄게 바뀜(계약 깨지는 지점).
6. **상태코드 제자리:** 데코레이터 status_code=성공(201), except 안=실패(400), 없음=404. `is None`으로 검사. service의 ValueError를 router가 400으로 번역(안 하면 500).
7. **디버깅:** 500의 진짜 원인은 터미널 트레이스백. greenlet은 async DB 다리(backend/.venv 사용). FK는 작성자 유저가 실재해야. BIGSERIAL은 누적 발급이라 id가 1부터가 아닐 수 있음.

# 세션 후반 — comments(평면 1:N) 4계층

> 위까지가 posts의 router를 단 부분. 이어서 **comments**를 같은 4계층(schema→repo→service→router)으로 한 번 더 만들었다. posts의 **축소 복습판**이라 새 개념은 적고, 대신 "데이터가 어디서 들어오나(입력 통로)"와 "없음=404 vs 입력틀림=400 구분"이 새로 박혔다.

## 전체 그림 — comments는 posts의 축소판

- Comment 모델: `id · post_id(FK posts) · author_id(FK users) · body · created_at`. 태그도 검색도 없는 **평면 1:N**(글 1개 ── 댓글 N개).
- 만든 파일 4개: `schemas.py`에 `CommentCreate`/`CommentOut` 추가, `repositories/comments.py`·`services/comments.py`·`routers/comments.py` 신규.
- 흐름은 posts와 동일: `router → service → repository → schema`. **새로 배운 건 딱 둘** — (1) 중첩 리소스 경로, (2) "없음"을 404로 번역하는 별도 예외.

# Part 10. 중첩 리소스 경로(nested route)

- **정의:** 댓글은 항상 특정 글에 속한다(`post_id` FK). 그래서 URL도 소속을 드러낸다:
  ```
  GET  /posts/{post_id}/comments   ← 그 글의 댓글 목록
  POST /posts/{post_id}/comments   ← 그 글에 댓글 작성
  ```
  여기서 `{post_id}`는 **부모(글)의 id**다. posts에서 `{post_id}`는 "그 글 자체"였지만, comments에선 "어느 글의 댓글이냐"는 **소속**을 가리킨다.
- **등장 배경:** `/comments?postId=5`로도 가능하지만, REST는 "댓글은 글에 종속된 자식 자원"이라는 관계를 **경로 계층**으로 표현하는 걸 선호한다(`/posts/5/comments`가 "5번 글의 댓글들"로 자명).
- **실제 런타임 장면:** `POST /posts/99999/comments`처럼 **없는 글**에 달면 부모가 없으니 **404**여야 한다(Part 11에서 처리).

# Part 11. 입력의 출처(location)가 다르다 — path / body / query / dependency

> "스키마 필드와 URL이 비슷하게 작용하네" (질문) — 여기서 FastAPI의 핵심 그림이 잡혔다.

**HTTP 요청 하나는 데이터가 여러 군데 흩어져 들어온다.** FastAPI는 "이 핸들러 인자를 **어디서** 꺼낼지"를 **타입·선언 방식으로 판별**해 자동으로 모아 한 함수의 인자로 합친다:

```python
@router.post("/posts/{post_id}/comments")
async def create_comment(
    post_id: int,                              # ① path  ← URL의 {post_id}
    body: CommentCreate,                       # ② body  ← 요청 JSON (Pydantic 모델)
    q: str | None = None,                      # ③ query ← ?q=... (있다면)
    session: AsyncSession = Depends(get_db),   # ④ 의존성 ← FastAPI가 주입
):
```

판별 규칙:
- **경로 변수(`{post_id}`)와 이름이 같은 단순 타입(int)** → **path**에서 꺼냄
- **Pydantic 모델 타입(`CommentCreate`)** → **요청 body(JSON)** 파싱
- 단순 타입 + 기본값 → **query string**(`?q=`)
- `Depends(...)` → **의존성** 주입

→ 그래서 "스키마 필드와 URL이 비슷"한 게 아니라, **둘 다 핸들러로 들어오는 입력인데 '통로(location)'가 다를 뿐**이다. body는 JSON 통로, path는 URL 통로. 이게 [개념] "타입 기반"의 실제 모습(타입 보고 어디서 꺼낼지 결정).

## 11.1 그래서 스키마에 뭘 넣고 뺐나

| 값 | 출처(통로) | CommentCreate에 넣나? |
|---|---|---|
| `body` | 요청 body (사용자 타이핑) | ✅ 넣음 |
| `post_id` | URL path | ❌ — path로 따로 받음 |
| `author_id` | 로그인 유저(B4, 지금 하드코딩) | ❌ — 서버가 채움 |

- `CommentCreate`에 `body`만 둔 건 **"post_id를 모른다"가 아니라** 사용자가 적는 값이 아니기 때문(URL에서 옴). 서버가 path에서 `post_id`를 꺼내 INSERT에 채운다.
- **`CommentOut`에 `post_id` 생략:** 호출자가 `GET /posts/5/comments`로 **이미 5번 글임을 알고** 부르니, 각 댓글에 `postId:5`를 또 박는 건 중복. → **원칙: 응답 모양은 그 자원의 접근 패턴(access pattern)을 따라간다.** 지금은 댓글을 글 안에서만(중첩) 보니 생략. 나중에 "전체 최근 댓글 피드"(평면 접근)가 생기면 그때 `post_id` 추가.
- `author_id`는 댓글마다 다르고 "누가 썼나" 표시에 쓰이니 응답에 **넣는다**(나중엔 username을 JOIN해 내려주는 게 더 좋음).

# Part 12. 없음(404) vs 입력 틀림(400) — 신호를 구분한다

> "404 not found 아닌가?" (질문) — 정확했다. 그런데 그냥 `ValueError`를 쓰면 404가 안 된다.

## 12.1 ValueError 하나로는 부족하다

posts의 router는 `except ValueError → HTTPException(400)`이다. comments service도 "post not found"에 `ValueError`를 던지면, 같은 패턴이 그걸 **400으로 번역**해버린다. 하지만 "글이 없음"은 **400(입력 틀림)이 아니라 404(자원 없음)**다.

| 상황 | 의미 | 상태코드 | 신호 |
|---|---|---|---|
| 없는 태그 slug | 입력이 틀림 | 400 | `ValueError` |
| 없는 글에 댓글 | 자원이 없음 | 404 | **별도 예외 필요** |

→ 두 실패는 **서로 다른 예외**로 신호해야 router가 구분해 번역한다.

## 12.2 service에 HTTPException(404)를 직접 쓰면 안 된다

"404니까 service에서 `raise HTTPException(404)`?" → **금지.** Part 4·6 원칙: **service는 HTTP를 모른다(404는 HTTP 개념).** service는 "**못 찾았다**"는 **도메인 신호**만 던지고, **404로 번역하는 건 router**다. 그래야 service를 HTTP 아닌 데서 재사용해도 안 깨진다.

## 12.3 선택: 내장 `LookupError`로 "없음"을 신호 (이번에 택함)

`LookupError` = 파이썬 내장 "찾기 실패" 예외(KeyError·IndexError의 부모). import 불필요.

```python
# services/comments.py
async def create(session, post_id, author_id, body):
    post = await get_post(session, post_id)
    if post is None:
        raise LookupError("post not found")     # ← 404로 번역될 신호
    comment = await create_comment(session, post_id=post_id, author_id=author_id, body=body)
    await session.commit()                       # 유스케이스 성공 → 확정
    return comment

# routers/comments.py
    try:
        comment = await service.create(session, post_id, author_id=1, body=body.body)
    except LookupError as e:
        raise HTTPException(status_code=404, detail=str(e))   # ← router가 404로 번역
```

이제 신호가 둘로 갈린다: `ValueError→400`(입력 틀림), `LookupError→404`(없음). (커스텀 `class NotFoundError(Exception)`로 가도 됨 — 더 명시적이고 재사용 좋지만, 이번엔 파일 안 늘리려 내장 LookupError 선택.)

> `body=body.body`가 어색해 보이지만 — 바깥 `body`는 **요청 객체**(CommentCreate), 안쪽 `.body`는 **댓글 본문 필드**. 같은 단어지만 층위가 다르다. 헷갈리면 인자명을 `payload`로 바꿔 `payload.body`.

# Part 13. `.asc()`는 호출해야 한다 — 정렬 표현식

```python
.order_by(Comment.id.asc())     # ✅
.order_by(Comment.id.asc)       # ❌
```

- **`Comment.id.asc`** (괄호X) = 아직 **실행 안 된 메서드 객체**(함수 그 자체).
- **`Comment.id.asc()`** (괄호O) = 그 메서드를 **호출해 나온 "정렬 표현식" 값**. `order_by`는 이 값을 받아 SQL `ORDER BY comments.id ASC`로 번역한다.
- **함수는 부를 때 결과가 나온다** — posts의 `desc()`와 같은 결, 세션 노트 "`b()` 호출해야 몸통이 돈다"와 동일. 괄호 빼면 정렬 기준으로 못 써서 500(또는 무시).
- 정렬 방향: 댓글은 `asc()`(오래된 것부터 = 대화 순서)가 자연스럽다. (posts 목록은 `desc()` = 최신 글 먼저였다.)

# Part 14. 실전 디버깅 — comments에서 만난 벽들

> 또 "돌려보면 안다". 작은 오타들이 연달아 막았고, 전부 **터미널 트레이스백 맨 아랫줄**이 범인을 가리켰다.

## 14.1 파일명 오타 — `commets.py` → `comments.py`
파일을 `commets.py`(n 빠짐)로 저장 → `from app.routers.comments import router`가 파일을 못 찾음. **import 경로는 실제 파일명과 글자까지 같아야** 한다.

## 14.2 대소문자 — `commentOut` → `CommentOut` (reload 실패 = "연결 끊김")
`response_model=list[commentOut]`(소문자 c) → 파이썬은 대소문자 구분 → `NameError: name 'commentOut' is not defined. Did you mean: 'CommentOut'?`.
**중요한 장면:** `uvicorn --reload`는 저장 때마다 앱을 다시 import하는데, import 중 에러가 나면 **앱 로딩 자체가 실패**해 서버가 안 뜬다 → 브라우저엔 **"연결 끊김/안 됨"** 으로 보인다. 즉 "연결 끊김"의 진짜 원인은 거의 항상 **터미널 트레이스백 맨 아랫줄**이다.

## 14.3 빈칸 미치환 — `Comment.___` → 500
repo의 `list_comments`에서 `___`를 안 채우고 둠 → `AttributeError: type object 'Comment' has no attribute '___'` → 500. (드릴의 `___`는 **지워야 할 자리표시**지 코드가 아니다.) 고친 것: `Comment.post_id == post_id`(왼쪽=컬럼, 오른쪽=함수 인자), `Comment.id.asc()`.

## 14.4 Swagger "Example Value" ≠ 실제 응답 (body:"string" 혼동)
`GET /posts/15/comments`가 200으로 댓글을 잘 돌려줬는데 `body`가 `"string"`이라 "내 댓글이 안 나오나?" 착각. 두 원인:
1. **POST할 때 Swagger 기본 예시값 `{"body":"string"}`을 안 바꾸고 보냄** → 본문이 글자 그대로 "string"으로 저장됨(버그 아님, 내가 보낸 값).
2. **두 박스 구분:** Swagger엔 박스가 둘 — **위쪽(content-length 있는 Server response)** = 방금 받은 **진짜 응답**, **아래쪽 "Example Value"** = 스키마 모양 예시(항상 `id:0, body:"string"`인 가짜). 헷갈리면 content-length 있는 위쪽을 본다.
실제 데이터인 증거는 `createdAt`이 행마다 다른 실제 시각이라는 것.

## 14.5 main.py — 두 라우터 등록 시 별칭(as)
posts·comments 라우터 변수명이 둘 다 `router`라 그냥 두 번 import하면 뒤엣것이 앞엣것을 덮어쓴다. **별칭으로 구분:**
```python
from app.routers.posts import router as posts_router
from app.routers.comments import router as comments_router
app.include_router(posts_router)
app.include_router(comments_router)
```

# Part 1. 비밀번호 해싱 — 왜 평문을 못 쓰나

## 1.1 정의 (용어부터)
- **plaintext(평문)**: 원본 그대로의 비밀번호(`"apple1234"`).
- **hash(해시)**: 입력을 고정 길이의 알아볼 수 없는 문자열로 바꾼 것. `$2b$12$Rk9...`.
- **one-way(단방향) function**: 입력→출력은 쉽지만 **출력→입력은 사실상 불가능**. "복호화"가 없다.
- **verify(검증)**: 저장 해시를 푸는 게 아니라, **입력 평문을 다시 해시해 저장 해시와 대조**.
- **salt(소금)**: 해싱 전 섞는 사람마다 다른 랜덤값. 같은 비번도 해시가 달라진다.
- **bcrypt**: 비밀번호 전용 해시. 일부러 느리고(cost) salt 내장.

## 1.2 등장 배경
평문 저장은 DB가 한 번 털리면 **모든 진짜 비번이 그대로 노출**된다(+ 재사용 피해 전파). → "서버조차 원본을 모르게" 저장하자 = 해싱. 단순·빠른 해시(MD5/SHA)는 초당 수십억 대입 + rainbow table에 뚫려서, **느리게(cost) + 사람마다 다르게(salt)** 만든 bcrypt가 표준이 됐다.

## 1.3 실제 런타임 장면
- 가입: `"apple1234"` → DB엔 `$2b$12$...`(60자). DB를 열어도 원본 모름.
- 로그인: 입력을 **또 해시**해 저장 해시와 비교. 두 사람이 같은 비번이어도 salt가 달라 해시는 서로 다르다.
- `verify`는 새 salt를 만들지 않고 **저장 해시에 박힌 salt를 꺼내 재사용** → 같은 비번이면 반드시 일치(안 그러면 영영 로그인 불가).

## 1.4 코드
```python
from passlib.context import CryptContext
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(pw: str) -> str:      return pwd_context.hash(pw)
def verify_password(pw, hashed) -> bool: return pwd_context.verify(pw, hashed)
```

## 1.5 막판 버그 — `bcrypt 5.0` × `passlib 1.7.4` 비호환 (500)
- 증상: signup이 `ValueError: password cannot be longer than 72 bytes`로 500. 비번은 4글자인데도.
- 원인: bcrypt 72바이트 제한을 옛 bcrypt는 잘라줬지만 5.0은 에러를 던짐 → passlib 1.7.4의 내부 자가테스트(긴 문자열 해시)가 깨짐.
- 해결: `requirements.txt`에 `bcrypt<4.1` 고정 후 재설치. **학습자 코드가 아니라 라이브러리 버전 충돌**이었다.

## 1.6 1차 출처
FastAPI *Security → "OAuth2 with Password (and hashing)"* / passlib `CryptContext`.

# Part 2. JWT — 로그인 증명을 토큰으로

## 2.1 정의
- **token(토큰)**: 로그인했음을 증명하는 서명된 문자열. 이후 요청마다 들고 온다.
- **JWT**: `header.payload.signature` 세 토막. header=서명 알고리즘, payload=담은 정보(claims: `sub`=주인, `exp`=만료), signature=비밀키로 서명한 값(위조 방지).
- **stateless(무상태)**: 서버가 "누가 로그인 중인지" 목록을 **안 들고 있음**. 토큰 자체가 증명이라 **서명만 검증**하면 됨.
- **encode/decode**: 만들기(payload+키→JWT) / 풀기(JWT+키→payload, 검증 실패 시 예외).

## 2.2 등장 배경 (세션 vs JWT)
옛 방식(세션)은 서버가 "세션id↔유저" 표를 저장하고 요청마다 조회 → 서버가 상태를 들고 있어야 하고 서버 여러 대면 표 공유 부담. JWT는 증명을 **토큰 안에 넣어 클라이언트가 들고 다니게** 해서 서버는 표 없이 서명만 검증 → 확장 쉬움.

## 2.3 실제 런타임 장면
- 로그인 성공 → `create_access_token`이 JWT 발급 → 프론트가 저장.
- 이후 요청 헤더 `Authorization: Bearer eyJ...` → 서버는 **서명·만료만 확인**(DB 세션 조회 없음).
- 토큰 1글자만 바꿔도 서명 검증 실패 → 401.

## 2.4 ⭐ HS256 ≠ bcrypt (헷갈린 지점)
| | bcrypt | HS256 |
|---|---|---|
| 일 | 비밀번호 **해싱**(단방향) | JWT **서명**(위조 방지) |
| 비밀키 | 없음(salt만) | 있음(`JWT_SECRET`) |
| 자리 | `hash_password` | `create_access_token` |

기억법: **bcrypt=비번을 숨김 / HS256=토큰에 도장.** 그래서 bcrypt엔 키가 없고 HS256엔 `JWT_SECRET`이 든다.

## 2.5 코드 (encode ↔ decode)
```python
# 발급 (security.py)
def create_access_token(sub: str) -> str:
    expire = datetime.now(timezone.utc) + timedelta(minutes=settings.access_token_expire_minutes)
    payload = {"sub": sub, "exp": expire}
    return jwt.encode(payload, settings.jwt_secret, algorithm="HS256")
# (jwt.encode가 exp의 datetime을 unix timestamp로 자동 변환)
```

## 2.6 막판 버그 — `config.py` 필드에 타입 어노테이션 누락
`jwt_algorism = 'HS256'`(어노테이션 X) → `PydanticUserError: non-annotated attribute`. Pydantic 필드는 **`이름: 타입 = 기본값`** 필수. + 오타 `algorism→algorithm`. 고침: `jwt_algorithm: str = "HS256"`.

## 2.7 1차 출처
FastAPI *Security: "...Bearer with JWT tokens"* / python-jose `jwt.encode/decode` / (배경) MDN HTTP *Authentication·Cookies*.

# Part 3. 인증 가드 — 토큰을 풀어 "현재 유저" 주입

## 3.1 정의
- **bearer token(소지자 토큰)**: "이 토큰을 가진 자"를 인증된 것으로 본다. `Authorization: Bearer <토큰>`.
- **`OAuth2PasswordBearer`**: 요청 헤더에서 `Bearer ` 뒤 토큰 문자열만 뽑아주는 FastAPI 도구. 없으면 자동 401. `tokenUrl`은 "로그인은 이 경로"라는 문서용 표시.
- **`Depends(...)`(의존성 주입)**: 핸들러 인자에 "이 값은 이 함수가 만들어 채워줘"라고 선언. 요청마다 그 함수를 먼저 실행해 결과를 끼워넣음.
- **의존성 중첩**: `get_current_user` 자신이 `Depends(get_db)`로 세션을 주입받는다(의존성이 의존성을 부름).

## 3.2 등장 배경
토큰 꺼내기·검증을 라우터마다 복붙하면 중복. → "현재 유저 만들기"를 **함수 하나**로 빼고, 보호 라우터는 `Depends(get_current_user)`만 붙인다(가드를 한 곳에 모음).

## 3.3 실제 런타임 장면
`user = Depends(get_current_user)`가 붙은 라우터는, 요청 도착 시 **핸들러 몸통 들어가기 전에** 가드를 먼저 돌린다. 토큰 없거나 틀리면 **거기서 401로 끊고 핸들러는 아예 실행 안 됨.** `/docs`엔 자물쇠/Authorize 버튼이 생긴다.

## 3.4 코드 — 필수 가드 + 선택적 가드
```python
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="auth/login")
oauth2_scheme_optional = OAuth2PasswordBearer(tokenUrl="auth/login", auto_error=False)  # 토큰 없어도 에러 X

async def get_current_user(token=Depends(oauth2_scheme), session=Depends(get_db)) -> User:
    try:
        payload = jwt.decode(token, settings.jwt_secret, algorithms=["HS256"])
        username = payload.get("sub")              # 발급 때 넣은 그 키 (encode↔decode 대칭)
    except JWTError:
        raise HTTPException(401, "invalid token")
    if username is None: raise HTTPException(401, "invalid token")
    user = await users_repo.get_by_username(session, username)
    if user is None:     raise HTTPException(401, "invalid token")
    return user

async def get_current_user_optional(...) -> User | None:   # 조회 라우트용: 실패 시 None
    if token is None: return None
    try: ... except JWTError: return None
    return await users_repo.get_by_username(session, username)
```
- **왜 두 종류냐**: 쓰기는 로그인 필수(`get_current_user`, 실패=401), **조회는 비로그인 허용**(`_optional`, 실패=None) → Part 7 myVote에서 쓴다.

## 3.5 막판 버그 — `Depends()` 빈칸 / `payload.get("oauth2_scheme")`
- `Depends()`(비움) → `Depends(oauth2_scheme)`. `oauth2_scheme`는 **도구 객체** → `Depends()` 안.
- `payload.get("oauth2_scheme")` → `payload.get("sub")`. `"sub"`는 **payload dict의 문자열 키**. 둘은 종류가 다름(객체 vs 키).

# Part 4. 회원가입·로그인 — 위 함수들을 처음 쓰는 곳

## 4.1 입력/출력 스키마 분리 ("평문은 들어오되 절대 나가지 않는다")
```python
class UserCreate(BaseModel):  username: str; email: str; password: str   # 입력: 평문 받음
class UserOut(BaseModel):     id: int; username: str; email: str          # 출력: 비번 흔적 0
                              # model_config = from_attributes=True
```
- 입력 모델을 응답에 재사용하면 해시/평문이 새어나갈 수 있다 → **다른 클래스**로. (B3 "입출력 분리"의 실물.)

## 4.2 회원가입 — 409 Conflict
- **409 Conflict**: 요청은 멀쩡한데 **현재 상태와 충돌**(이미 있는 username/email). `users.username·email`이 DB UNIQUE.
- 패턴: `get_by_username`/`get_by_email`로 **미리 조회 → 있으면 409**. DB UNIQUE는 마지막 안전망(없으면 IntegrityError→500으로 노출).
```python
@router.post("/signup", response_model=UserOut, status_code=201)
async def signup(body, session=Depends(get_db)):
    if await users_repo.get_by_username(session, body.username): raise HTTPException(409, "username taken")
    if await users_repo.get_by_email(session, body.email):       raise HTTPException(409, "email taken")
    hashed = hash_password(body.password)   # ← 막판 버그: 내장 hash()를 썼었다(정수·무작위) → hash_password
    user = await users_repo.create_user(session, username=..., email=..., password_hash=hashed)
    await session.commit(); return user
```

## 4.3 로그인 — OAuth2 폼 + 401
- **`OAuth2PasswordRequestForm`**: OAuth2 표준이라 로그인은 JSON이 아니라 **폼**(`username`/`password`)으로 받는다 → `/docs`의 Authorize와 자동 연동. (`python-multipart` 필요 — 막판에 설치.)
- 유저 없음/비번 틀림을 **둘 다 401**로 뭉뚱그림(있는지 없는지를 공격자에게 안 흘림).
```python
@router.post("/login", response_model=Token)
@limiter.limit("5/minute")                       # Part 6
async def login(request: Request, form: OAuth2PasswordRequestForm = Depends(), session=Depends(get_db)):
    user = await users_repo.get_by_username(session, form.username)
    if user is None or not verify_password(form.password, user.password_hash):
        raise HTTPException(401, "invalid credentials")
    return Token(access_token=create_access_token(sub=user.username))
```

## 4.4 막판 버그 2개 (라우터 등록·정의 순서)
- `app.include_router(auth.router)` ← `auth` 미import(별칭은 `auth_router`) → `NameError`. → `auth_router`.
- 로그인 함수를 `router = APIRouter(...)` **위**에 붙임 → 데코레이터 `@router.post`가 아직 없는 `router`를 참조 → `NameError`. **데코레이터는 router 정의 뒤에.**

# Part 6. CORS·rate limit

## 6.1 CORS
- **정의**: 브라우저는 보안상 *다른 origin*(프론트 `:5173` → 백엔드 `:8000`)으로의 요청을 기본 차단(Same-Origin Policy). `CORSMiddleware`가 "이 origin 허용" 헤더를 붙여 풀어줌.
- 없으면 프론트 fetch/axios가 **CORS 에러**로 막힌다. → Part 8 프론트 연결의 전제.
```python
app.add_middleware(CORSMiddleware, allow_origins=["http://localhost:5173"],
                   allow_credentials=True, allow_methods=["*"], allow_headers=["*"])
```

## 6.2 rate limit (slowapi)
- **정의**: 단위 시간당 호출 횟수 제한. 로그인 **brute-force(비번 무차별 대입)** 방어. 초과 시 429.
- `key_func=get_remote_address`(요청자 IP 기준). 데코레이터 `@limiter.limit("5/minute")` + 엔드포인트에 `request: Request` 필수.
- `main.py`에 `app.state.limiter` + `RateLimitExceeded` 핸들러 등록.

# Part 7. myVote — "내가 이 글에 어떻게 투표했나"

## 7.1 왜 필요 / 없으면 어떻게 보이나
- `score`(net 집계)는 "이 글 총점"이라 **누가 봐도 같다**. `myVote`는 "**내**가 +1/-1/0"이라 **보는 사람마다 다르다** → 현재 로그인 유저를 알아야 계산.
- 없으면: 로그인했어도 목록을 다시 열면 **자기가 어디에 투표했는지 표시가 사라진다**(버튼 하이라이트 불가). 비로그인 문제가 아니라 **로그인 UX 문제**.

## 7.2 선택적 인증으로 "조회는 누구나, 표시는 로그인하면"
- GET 라우터에 `Depends(get_current_user_optional)` → 토큰 있으면 `user.id`, 없으면 `None`.
- `user_id=None`이면 myVote=0(비로그인), 있으면 그 유저의 투표를 얹는다. **글 보기 자체는 안 막힌다.**

## 7.3 N+1 방지 — 목록은 한 번의 일괄 조회
```python
# 상세: 한 글의 내 표 1건
async def _my_vote(session, user_id, post_id) -> int:
    if user_id is None: return 0
    stmt = select(Vote.value).where(Vote.user_id==user_id, Vote.target_type=="post", Vote.target_id==post_id)
    return (await session.execute(stmt)).scalar_one_or_none() or 0

# 목록: 글마다 따로 쿼리하면 N+1 → 한 번에 in_()로 모아 dict 매핑
vstmt = select(Vote.target_id, Vote.value).where(Vote.user_id==user_id, Vote.target_type=="post", Vote.target_id.in_(ids))
vote_map = {tid: val for tid, val in (await session.execute(vstmt)).all()}
for p in items: p.my_vote = vote_map.get(p.id, 0)
```
- `PostOut`에 `my_vote: int = Field(default=0, alias="myVote")` 추가. ORM 객체에 `p.my_vote`를 얹으면 `from_attributes`가 읽어 응답에 나간다.
- **N+1 문제**: "목록 N개 + 각자 1쿼리 = N+1번"의 낭비. 미리 한 번에 모아 메모리에서 매핑.

# Part 8. 프론트 mock → 실제 HTTP (api 계층)

## 8.1 ⭐ 본질: "기억 → 네트워크" (async는 그대로)
- 전(mock): `fetchPosts`가 `MOCK_POSTS` **배열에서 꺼냄**(같은 메모리). 후: **다른 프로세스(백엔드)에 네트워크 요청** → 진짜 DB.
- `async/await`는 **둘 다 있었다**. 바뀐 건 "기다리는 방식"이 아니라 **"누구에게 묻느냐"**. (비유: 주머니 메모 vs 식당에 전화.) → "비동기 이미 있는데?"의 답.

## 8.2 fetch vs axios — 왜 fetch로 본질을 봤나
- axios는 "본질을 이미 가린 완성품". **fetch로 래퍼를 직접 짜보면** axios가 대신 해주던 일이 드러난다.
- `api/http.ts`(직접 만든 래퍼)가 하는 5가지 = axios가 자동으로 하는 5가지:
  1. baseURL 붙이기 2. **토큰 자동 첨부**(localStorage→`Bearer`) 3. 본문 인코딩(객체→JSON / 로그인=폼) 4. **status 확인 후 throw** 5. `res.json()` 파싱
- ⭐ 4번: **fetch는 404·500에도 에러를 안 던진다**(네트워크는 성공이니까). `if (!res.ok) throw`를 직접 넣어야 함 — axios는 이걸 기본 제공. → "왜 axios를 쓰나"가 여기서 보임.

## 8.3 경계에서의 번역 — toPost
- 백엔드 JSON과 프론트 `Post`는 모양이 다름(id 숫자 vs 문자열, score/myVote 유무). `toPost`가 **api 함수 안에서** 변환 → 컴포넌트는 깨끗한 `Post`만 받음. (AGENTS "프론트가 본문만 교체" + "매핑은 경계에서 1번만"의 실물.)

## 8.4 토큰 저장 — localStorage
- 로그인 응답의 `access_token`을 `localStorage`에 저장 → `http.ts`가 모든 요청에 자동 첨부. 새로고침해도 유지(AuthContext가 `localStorage`에서 초기화), logout 시 제거.

## 8.5 프론트 서버(Vite) vs 백엔드 서버 — 다른 일
| | 백엔드(uvicorn :8000) | 프론트(Vite :5173) |
|---|---|---|
| 일 | DB 조회·JSON 응답 | TSX/모듈 → JS **즉석 컴파일** + 핫리로드 + 파일 제공 |
| 왜 | 데이터가 거기 | 브라우저는 `.tsx`·`import`를 직접 못 읽음 |
- 브라우저는 5173에서 **페이지(HTML+JS)를 받고**, 그 JS가 다시 8000으로 **fetch** → 출처가 달라 **CORS 필요**(Part 6).
- 배포: `npm run build`로 **정적 파일** 생성 → 정적 호스팅(Vite 프로세스 불필요). 백엔드는 배포에서도 프로세스로 떠 있어야 함.

# 디버깅 로그 (이번에 막힌 것 모음 — 다 "코드 밖" 또는 "이름/순서")
1. **signup 500** = `bcrypt 5.0`×`passlib 1.7.4` 비호환 → `bcrypt<4.1`. (라이브러리 버전)
2. **`hash()` 내장함수** 사용 → `hash_password`. (이름 혼동, 정수·무작위라 위험)

4. **`include_router(auth.router)`** = NameError(별칭은 `auth_router`).
5. **로그인 함수가 `router` 정의 위** = NameError(데코레이터는 정의 뒤).
6. **`python-multipart` 미설치** = OAuth2 폼 RuntimeError → 설치.
7. **`Depends()` 빈칸 / `payload.get("oauth2_scheme")`** = 도구 객체 vs payload 키 혼동.
> 교훈: B4의 막힘은 개념보다 **이름·정의 순서·버전**이 대부분. "에러 메시지를 그대로 읽기"가 제일 빠른 길이었다.

# 한 장 요약

- **해싱**: 단방향 저장. `verify`는 복호화가 아니라 "다시 해시해 대조"(저장 해시의 salt 재사용). bcrypt=느림+salt. (버전: `bcrypt<4.1`.)
- **JWT**: `header.payload.signature`. stateless(서버가 세션표 안 듦), 서명만 검증. **HS256(토큰 서명, 키 있음) ≠ bcrypt(비번 해싱, 키 없음).** encode↔decode 대칭(`sub`).
- **가드**: `Depends(get_current_user)`가 핸들러 전에 돌아 토큰 검증→유저 주입(실패 401). 조회용 `_optional`은 실패 시 None(비로그인 허용).
- **signup/login**: 입출력 스키마 분리(비번 안 나감)·중복 409·로그인은 OAuth2 **폼**+401(유저없음/비번틀림 동일 메시지).
- **보호+하드코딩 제거**: 쓰기에 가드, `author_id=1`→`current_user.id`. 조회는 공개.
- **CORS**(다른 origin 허용)·**rate limit**(로그인 5/min, brute-force).
- **myVote**: 선택적 인증 + 일괄 조회(N+1 방지). 비로그인=0, 로그인=내 표. `PostOut.myVote`.
- **프론트 실제 HTTP**: mock(배열)→fetch(네트워크), async는 원래 있던 것. fetch 래퍼=axios의 본질(특히 4xx도 안 throw → 직접 확인). toPost로 경계 번역. 토큰 localStorage. Vite(코드 서빙) ≠ 백엔드(데이터).
- **상태: 백엔드 B0~B4 종료.** 남은 건 프론트 8·9 재드릴(로그인폼·ProtectedRoute 등; api는 참고 구현 있음).

### `frontend/src/pages/PostDetailPage.tsx` — 신규

```tsx
import {useState, useEffect} from "react";
import {useParams} from "react-router-dom";
import {fetchPostById} from "../api/posts";
import type {Post} from "../types/post";

function PostDetailPage() {
    const {id} = useParams();
    const [post, setPost] = useState<Post | null>(null);
    const [loading, setLoading] = useState(true);
    const [error, setError] = useState<string | null>(null);

    useEffect(() => {
        async function load() {
            try {
                const data = await fetchPostById(id!);
                setPost(data);
            } catch {
                setError("글을 불러오지 못했습니다");
            } finally {
                setLoading(false);
            }
        }
        load();
    }, [id]);

    if (loading) return <p className="p-8">불러오는 중...</p>;
    if (error) return <p className="p-8 text-red-600">{error}</p>;
    if (!post) return null;

    return (
        <div className="mx-auto max-w-2xl px-4 py-8">
            <h1 className="mb-4 text-2xl font-bold">{post.title}</h1>
            <p>{post.excuseText}</p>
        </div>
    );
}

export default PostDetailPage;
```

## 1. 동적 라우트 세그먼트 `:id`

상세 화면은 글마다 URL이 달라야 한다 — `/posts/p1`, `/posts/p2`, … 글 개수만큼 `<Route>`를 쓸 순 없다. 그래서 path에 **변수 자리**를 둔다:

```tsx
<Route path="/posts/:id" element={<PostDetailPage/>} />
```

- 콜론 `:`은 "이 자리는 고정 글자가 아니라 **아무 값이나 들어올 수 있는 칸**이고, 그 값을 `id`라는 이름으로 부르겠다"는 표시.
- `/posts/p1` → `id = "p1"`, `/posts/p99` → `id = "p99"`.

### ⚠️ 이번 세션 버그: 콜론 빠뜨림

```tsx
<Route path="/posts/id" ... />    // ❌ id가 변수가 아니라 그냥 글자 "id"
<Route path="/posts/:id" ... />   // ✅
```

콜론이 없으면 이 라우트는 오직 URL이 **정확히 `/posts/id`** 일 때만 매칭된다. `/posts/p1`엔 안 맞고 → `useParams()`가 `{}`(빈 객체)를 줘서 `id`는 `undefined` → `fetchPostById(undefined!)` → 글을 못 찾는다. **빨간 줄(타입 에러)은 안 뜨지만 동작이 안 되는** 종류의 버그라 더 주의.

---

## 2. `useParams` — URL의 변수 자리 꺼내기

컴포넌트 안에서 그 `id` 값("p1")을 어떻게 꺼낼까? `react-router-dom`의 `useParams()` 훅이 한다.

- 호출하면 **URL의 변수 자리들을 모은 객체**를 돌려준다. path가 `/posts/:id`였으니 `{ id: "p1" }`.
- 그래서 `const {id} = useParams();`로 구조분해해 꺼낸다.
- **타입이 `string | undefined`** 다. 라우터는 "혹시 그 자리가 비었을 수도 있다"고 보기 때문. 그래서 `fetchPostById(id!)`처럼 `!`(non-null assertion, "내가 보증한다 null 아니다")를 붙이거나, `if (!id) return` 같은 가드로 한 번 짚어줘야 타입 검사기가 통과시킨다.

## 3. 로드 패턴 재사용 — 딱 한 군데만 다르다

"화면 뜰 때 한 번 비동기로 데이터 가져오기"는 세션7 전반부(`PostListPage`)에서 한 그대로다:

```
useEffect( () => {
    async function load() { try { await … } catch {…} finally { setLoading(false) } }
    load();
}, [의존성] )
```

`PostListPage`와 **다른 점은 의존성 배열 하나**:

| | PostListPage | PostDetailPage |
|---|---|---|
| 의존성 배열 | `[]` | `[id]` |
| 의미 | "처음 한 번만" | "처음 + id 바뀔 때마다" |

### 왜 `[id]`인가 (인과 먼저)

URL의 id가 `p1` → `p2`로 바뀌면 **다른 글을 다시 불러와야** 한다. 그래서 "id가 바뀌면 effect를 다시 실행하라"고 알려준다. 만약 `[]`로 두면 첫 글(`p1`)만 잡고, 이후 `p2`로 이동해도 effect가 재실행되지 않아 **화면이 옛 글에 머문다**.

> 의존성 배열의 의미 복습:
> - `[]` → 마운트 시 1회.
> - `[id]` → 마운트 시 + `id`가 바뀔 때마다.
> - 생략 → 매 렌더마다 (대개 버그).

### `Post | null` 초기값

`const [post, setPost] = useState<Post | null>(null);`
- 아직 글을 못 받았을 때를 표현하려고 `null` 초기값.
- 그래서 렌더에서 `if (!post) return null;` 가드가 필요(로딩도 에러도 아닌데 아직 post가 안 채워진 찰나).

### ⚠️ 이번 세션 에러 1: `useState`를 호출 안 함

```tsx
const [post, setPost] = useState<Post | null>;     // ❌ TS2488
const [post, setPost] = useState<Post | null>(null);// ✅
```

`const [a, b] = X` 구조분해는 X가 **배열**이어야 한다(배열만 `[Symbol.iterator]`를 가짐). 괄호 `()`를 빼면 X 자리에 들어간 게 **`useState` 함수 그 자체**(배열 아님)라 "iterator가 없다"고 막는다. `useState`는 **호출해야** 비로소 `[현재값, setter]` 배열을 돌려준다.

### ⚠️ 이번 세션 에러 2: 컴포넌트 이름 소문자

```tsx
function postDetailPage() {…}   // ❌ 소문자 시작
function PostDetailPage() {…}   // ✅
```

컴포넌트 함수는 **반드시 대문자 시작**(`PostCard`에서 배운 규칙). 소문자면 React가 HTML 태그로 취급한다. default export라 import는 안 깨지지만 규칙 위반.

### ⚠️ 이번 세션 에러 3: 파일 통째 중복

`PostListPage.tsx`에 `function PostListPage()`가 두 번 들어가 `TS2393: Duplicate function implementation`. 같은 스코프에 같은 이름 함수 두 번 선언 불가 → 복제본 블록 삭제로 해결.

### 스타일: 함수 선언문 뒤 세미콜론 불필요

`function load() { … }`는 **함수 선언문**이라 끝에 `;`가 필요 없다. (`;`는 `const x = …;` 같은 **문**에 붙는다.)

## 4. `<Link>` vs `<a>` — SPA에서 이동하는 법

목록에서 글을 누르면 상세로 가야 한다. 그런데 라우트만 있고 **그 URL로 가는 길**이 화면에 없으면, 주소창에 직접 `/posts/p1`을 쳐야만 보인다. 그 길이 `<Link>`다.

- HTML 기본 이동은 `<a href="...">`. 하지만 SPA에서 `<a>`를 누르면 브라우저가 **서버에 새 HTML을 통째로 다시 요청** → 화면 깜빡 새로고침 + React 앱 처음부터 재시작 + 메모리의 state 전부 소실.
- `react-router-dom`의 `<Link to="...">`는 겉보긴 같지만, 서버에 다시 요청하지 않고 **URL만 바꾼 뒤 그 URL에 맞는 Route 컴포넌트로 화면만 갈아끼운다**. 새로고침 없이, state 유지한 채로.

### `to` 속성과 템플릿 리터럴

- `<a>`가 `href`로 목적지를 받듯, `<Link>`는 **`to`** 로 받는다.
- 글마다 id가 다르니 `"p1"`을 고정으로 박으면 안 되고, 그 카드가 받은 `post.id`를 끼워 넣는다 → **템플릿 리터럴**(백틱) 사용:
  ```ts
  `/posts/${post.id}`   // post.id가 "p1"이면 → "/posts/p1"
  ```
- JSX 속성 안에서는 **중괄호로 감싼 뒤** 그 안에 백틱을 쓴다: `to={`/posts/${post.id}`}`. (중괄호 = "여기부터 JS")

### ⚠️ 이번 세션 버그 2개 (PostCard)

1. **중괄호 누락 (문법 에러):**
   ```tsx
   <Link to=`/post/${post.id}`>     // ❌ =뒤 백틱이 맨몸 → 파서 막힘
   <Link to={`/posts/${post.id}`}>  // ✅
   ```
   JSX 속성에 JS 표현식을 넣을 땐 반드시 `{}`로 감싼다. `to="..."`=고정 문자열, `to={...}`=JS.

2. **경로 단/복수 불일치 (논리 버그, 빨간 줄 없음):** 라우트는 `/posts/:id`(복수)인데 `<Link>`는 `/post/...`(단수)로 보내 매칭 실패 → 클릭하면 빈 화면. 철자 1글자 차이도 매칭 안 됨.

## 5. ⭐ 클라이언트 사이드 라우팅 — 등장 배경·원리·state 보존

### 5-1. 등장 배경: 서버 라우팅의 한계

원래 웹: 브라우저가 `/posts/p1`을 **서버에 요청** → 서버가 그 페이지 HTML을 **통째로 만들어 응답** → 브라우저가 받은 HTML로 화면을 **처음부터 다시 그림**. 링크 누를 때마다 이 왕복 반복 = **서버 사이드 라우팅**("어떤 URL에 어떤 화면"의 결정권이 서버에).

문제 둘:
- **느리고 깜빡인다.** 일부만 바뀌어도(목록→상세, 헤더·메뉴는 그대로) 매번 HTML 전체를 새로 받아 백지부터 다시 칠함 → 흰 깜빡임 + 네트워크 대기.
- **앱 상태가 매번 초기화.** 페이지를 새로 받으면 JS가 처음부터 재실행 → 입력 중이던 폼, 스크롤 위치, 메모리의 데이터 전부 리셋.

웹이 "문서 묶음"에서 "앱"(Gmail 같은)으로 진화하며 이 깜빡임·초기화가 견딜 수 없게 됨 → **SPA + 클라이언트 사이드 라우팅** 등장: HTML은 **딱 한 번만** 받고, 이후 화면 전환은 **브라우저 안의 JS가 직접** 처리.

### 5-2. 원리 — 두 기둥

**(A) History API:** 브라우저가 주는 JS 함수 `history.pushState(state, "", "/posts/p1")` — **주소창 URL을 바꾸지만 서버에 요청하지 않는다**(주소만 갈아끼움). "뒤로 가기"를 누르면 `popstate` 이벤트 발생. `react-router`는 이 둘(`pushState` + `popstate` 구독)을 내부에서 감싼다.

`<Link>` 클릭 시 실제 일어나는 일:
1. `<Link>`가 브라우저 기본 동작(`<a>`의 서버 요청)을 **막는다**(`event.preventDefault()`).
2. 대신 `history.pushState`로 **URL만** `/posts/p1`로 바꾼다(서버 요청 0회).
3. react-router가 URL 변경을 감지 → `<Routes>` 안에서 현재 URL에 맞는 `<Route>`를 찾아 그 element(`<PostDetailPage/>`)를 렌더 대상으로 고른다.
4. React가 **바뀐 부분만** 화면에 다시 그린다.

**(B) state가 유지되는 원리:**

먼저 state가 어디 사는지 — React state(`useState` 값들)는 **브라우저 탭의 JS 메모리(힙)** 안, React가 관리하는 트리에 있다. DOM(화면)이 아니라 JS 메모리.

핵심: **state를 날리는 건 "URL 변경"이 아니라 "JS 프로그램의 재시작"이다.**

- **서버 라우팅(`<a>`):** 새 HTML을 받으면 브라우저가 기존 페이지를 파괴하고 새 문서를 띄움 → JS 실행 환경(메모리·힙) 전체가 버려지고 새로 시작 → React 앱도 처음부터 재실행 → 모든 `useState`가 초기값으로 재생성. **메모리가 통째로 갈렸으니** state 소실이 당연.
- **클라이언트 라우팅(`<Link>`):** 새 문서를 안 받음 → 같은 JS 프로그램이 끊김 없이 계속 실행, 메모리도 그대로 → 바뀐 건 URL 문자열 + React가 고른 "보여줄 컴포넌트"뿐 → 그 라우팅과 **무관한** 컴포넌트(헤더, 전역 Context, 다른 state)는 메모리에 그대로 살아있음. React는 화면을 다시 칠할 때 변경분만 반영하지 메모리를 비우지 않음.

**한 줄 요약:** `<a>` = "프로그램 종료 후 재실행"이라 메모리 리셋 / `<Link>` = "실행 중 프로그램이 화면만 바꿈"이라 메모리(=state) 보존.

> ⚠️ 구분: `PostDetailPage`는 이동마다 **새로 마운트**되므로 그 안의 `post`/`loading` state는 매번 새로 시작한다(컴포넌트가 새로 생기니까). "유지된다"는 건 **언마운트되지 않고 계속 떠 있는** 컴포넌트들(레이아웃·전역 상태) 얘기. 라우팅이 "앱 전체 리셋"이 아니라 "필요한 가지만 교체"라는 게 핵심.

## 6. `<h2>` 안에 `<Link>`를 넣는 이유

**의미(semantic)와 기능(behavior)을 분리해 둘 다 챙기는 구조.**

- `<h2>` = "이 텍스트는 **제목**이다"라는 **의미** 담당. 화면 리더기·검색엔진·문서 구조 분석이 "여긴 제목"이라 인식하는 건 `<h2>` 덕분. 글자 키우려는 게 아니라 **문서의 뼈대**를 알려주는 태그.
- `<Link>` = "누르면 **이동한다**"는 **기능** 담당.

둘을 **겹쳐서** "제목이면서 동시에 클릭하면 이동하는 것"을 만든다. 바깥 `<h2>`라 의미는 '제목'으로 유지, 안의 `<Link>`가 클릭 동작을 입힘.

**왜 `<h2><Link>` 순서이고 `<Link><h2>`가 아닌가:** 바깥쪽="이게 무엇인가(정체성)", 안쪽="그 텍스트에 걸리는 동작". `<h2>` 안에 인라인 요소(`<Link>`→`<a>`)는 HTML 표준에 맞음(제목 안 링크는 흔함). 반대로 `<a>` 안에 블록 제목보다 구조가 깔끔.

**대안 비교:** `<h2 onClick={…}>`로 제목에 직접 클릭을 달 수도 있지만 → 키보드 접근 불가, 우클릭 "새 탭으로 열기" 불가, 호버 시 목적지 URL 미표시. `<Link>`(실제 `<a>`로 렌더)는 이 **링크다운 기본기**가 공짜.

## 7. 헷갈리기 쉬운 구분

- `path="/posts/:id"`(콜론=변수 칸) vs `path="/posts/id"`(글자 "id" 고정)
- `useParams()` 반환 `{id}`의 타입 = `string | undefined`(그래서 `!` 또는 가드 필요)
- `[]`(처음 1회) vs `[id]`(처음 + id 변경마다)
- `<a>`(서버 재요청, 메모리 리셋) vs `<Link>`(URL만 변경, state 보존)
- `to="..."`(고정 문자열) vs `to={...}`(JS 표현식)
- `useState<T>`(호출 안 함 → 함수 자체) vs `useState<T>(초기값)`(호출 → `[값, setter]` 배열)

---

## 8. 한 줄 요약 모음

- 글마다 다른 URL은 `path="/posts/:id"`의 콜론 변수로 받고, 컴포넌트에선 `useParams()`로 꺼낸다.
- 상세 페이지 로드 패턴은 목록과 같고, **의존성만 `[id]`** — id가 바뀌면 다시 불러오기.
- SPA 내부 이동은 `<a>`가 아니라 `<Link to={…}>` — 서버 재요청 없이 URL만 바꿔 화면을 갈아끼우므로 빠르고 state가 보존된다.
- state는 DOM이 아니라 JS 메모리에 살고, `<a>`는 그 메모리를 통째로 리셋하지만 `<Link>`는 건드리지 않는다.
- 제목을 누르면 이동하게 하려면 `<h2>`(의미) 안에 `<Link>`(기능)를 넣는다.

## 심화 Q&A (이번 세션 추가 질문 4개 — 이번 주 범위 밖, 개념만)

> 세션7-2 본문(단발성 useEffect fetch)에서 파생된 질문들. 실습은 안 했지만 원리는 배운 것(setter 통보·리렌더·VDOM diff)의 연장선이라 복습용으로 묶어둔다.

### Q1. 서버에서 "실시간으로" 받아와야 하는 데이터는?

우리 `useEffect` fetch는 **단발성 스냅샷** — 마운트 시(또는 `id` 변경 시) 한 번만 묻고 끝. 그 뒤 서버 데이터가 바뀌어도 화면은 모른다. 실시간이 필요하면 "서버의 변화를 어떻게 다시 가져오나"가 추가돼야 하고, 방법은 둘:

**(A) 폴링(polling) — 클라이언트가 주기적으로 다시 묻기.** `setInterval`로 "N초마다 다시 fetch".
```tsx
useEffect(() => {
    const tick = () => fetchPostById(id!).then(setPost);
    tick();
    const timer = setInterval(tick, 5000);
    return () => clearInterval(timer);   // 언마운트 시 정리(cleanup)
}, [id]);
```
- 장점: 평범한 REST면 그대로 됨(특별 인프라 불필요). 단점: 변화 없어도 계속 묻고(낭비), 최대 N초 지연. "거의 실시간"이면 충분한 곳(대시보드)에 적합.

**(B) 푸시(push) — 서버가 변화를 먼저 밀어주기.** 연결을 계속 열어두고 서버가 바뀔 때마다 전송(진짜 실시간).
- **WebSocket:** 양방향 상시 연결(채팅처럼 양쪽이 말해야 할 때).
- **SSE(Server-Sent Events):** 서버→클라이언트 단방향 스트림(알림·피드처럼 서버가 일방적으로 밀어줄 때, 더 가벼움).

**공통점:** 데이터를 어떻게 가져오든, 화면 반영의 마지막 단계는 **항상 `setState`**. "useEffect fetch = 사진 한 장 / 실시간 = 동영상", 동영상은 폴링(주기적 재촬영) 또는 푸시(서버가 프레임 전송)로 만든다. 실무에선 TanStack Query(자동 재요청·캐싱)나 WebSocket 클라이언트를 쓰지만 밑바닥 원리는 위 둘.

### Q2. state는 실제로 어떤 형태로 저장되나? (우리 코드로)

**오해부터:** state는 `post`/`loading`/`error`라는 **변수 이름으로 저장되지 않는다.** 그 이름들은 매 렌더마다 새로 생기는 지역 const일 뿐. 실제 저장은 **호출 순서**로 관리된다.

React는 화면에 떠 있는 컴포넌트마다 **fiber**(JS 메모리의 내부 객체)를 만들고, 거기에 hook들을 **호출된 순서대로 연결 리스트(linked list)**로 매단다. `useState`마다 hook 노드 하나가 생기고, 그 노드의 `memoizedState` 칸에 **현재 값**이 산다. `PostDetailPage`(useState 3 + useEffect 1)를 글 다 불러온 상태로 그리면:

```js
fiber (PostDetailPage)
 └─ memoizedState ─▶ hook#0 ─next─▶ hook#1 ─next─▶ hook#2 ─next─▶ hook#3 ─▶ null
                     (post)          (loading)      (error)        (useEffect)

hook#0 = { memoizedState: { id:"p1", title:"지각 변명", ... }, queue:{pending:null}, next:hook#1 }  // post
hook#1 = { memoizedState: false, queue:{pending:null}, next:hook#2 }                                 // loading
hook#2 = { memoizedState: null,  queue:{pending:null}, next:hook#3 }                                 // error
hook#3 = { memoizedState: { deps:["p1"], destroy:undefined }, next:null }                            // useEffect
```
(내부 필드명은 더 많고 다르지만 구조는 정확히 이것: fiber에 매달린 hook 리스트, 각 노드 `memoizedState`가 값.)

**리렌더 시:** `PostDetailPage()`가 다시 호출되면 `useState`를 만날 때마다 React가 "이번 컴포넌트의 0번째/1번째/2번째 hook"을 순서대로 꺼내 그 `memoizedState`를 지역 변수에 넣어줌(초기값 `null`은 값이 이미 있으면 무시). 즉 `post`/`loading`/`error`는 **순번표 보고 리스트에서 값 받아오는 지역 변수**, 값 자체는 fiber의 hook 노드에 산다.

**⭐ "hook을 조건문 안에서 부르면 안 되는" 이유:** 값을 순서(0,1,2번)로 찾으니, 어느 렌더에서 `if`로 `useState` 하나를 건너뛰면 순번이 밀려 `loading`이 받을 값을 `error`가 받아버린다. 그래서 hook은 항상 같은 순서·최상단에서. (`react-hooks/rules-of-hooks` 린트가 잡음.)

### Q3. state가 "바뀌었다"는 걸 어떻게 인식하나?

**핵심: React는 state 변수를 감시하지 않는다. 변화를 *알아채는* 게 아니라 setter로 *통보*받는다.**

**층위 1 — 감시(polling)가 아니라 통보(push).** 메모리를 주기적으로 스캔하지 않음. `setPost(data)` **호출 자체가 신호**. 그래서 `post = data` 직접 대입은 화면이 안 바뀐다(통보 없음). (cf. Angular 옛 dirty-checking = polling / React = setter push.)

**층위 2 — setter 내부 동작.** `setPost`는 그 컴포넌트의 fiber·hook#0을 클로저로 붙든 함수. 호출하면 ① hook#0의 `queue.pending`에 다음 값 등록 ② fiber를 'dirty'로 표시하고 리렌더 예약(`scheduleUpdateOnFiber`) ③ 잠시 후 컴포넌트 재호출하며 큐 비워 `memoizedState` 갱신. 즉 "인식"의 실체 = **setter가 fiber에 플래그 세우고 재렌더 예약**(감지 로직이 아니라 등록 로직).

**층위 3 — "진짜 바뀐 값인가" 판단 = `Object.is` 참조 비교.** setter 불렸다고 무조건 안 그림. 새 값 vs 현재 값을 `Object.is`(거의 `===`, 참조 동일성)로 비교 → 같으면 리렌더 건너뜀(bail out). **`Object.is`는 참조(주소)를 비교하지 내용을 안 봄:**
```ts
posts.push(newPost); setPosts(posts);  // 같은 주소 → "안 바뀜" → 스킵! ❌
setPosts([...posts, newPost]);          // 새 주소 → "바뀜" → 리렌더 ✅
```
배열 내용이 달라져도 **제자리 수정은 주소가 안 변해** 동일로 판정 → 그래서 state는 **불변(immutable)하게 새 객체/배열로 교체**(세션6 push 파괴적 vs concat/스프레드 비파괴적과 연결). (`fetchPostById`가 매번 새 객체를 주는 경우는 자동으로 주소가 달라 OK. 주의할 건 기존 state 직접 수정 후 재투입.)

**층위 4 — "화면의 무엇을 바꿀지"는 별도 단계.** 위는 "다시 렌더할지" 결정. 렌더하면 새 VDOM 트리를 만들고 이전 VDOM과 **diff(reconciliation)** 해서 실제 DOM 중 바뀐 부분만 고침.

요약: **인식 = 감시가 아니라 setter 통보 → fiber 플래그+리렌더 예약 → `Object.is` 참조 비교로 거름(불변성 필수) → VDOM diff로 DOM 반영.**

### Q4. 한 요소 값만 실시간으로 끊김없이 갱신(주식창)? React로 가능?

**가능하고 React 강점.** 두 가지 조합: **(A) 계속 받아오는 연결 + (B) 리렌더를 그 한 요소로 가두기.**

**(A) 데이터 = 푸시 연결.** 호가는 폴링으로 부족 → WebSocket/SSE로 서버가 틱마다 밀어줌. "끊김없이 계속"의 정체는 무한 루프가 아니라 **이벤트 구동**(메시지 도착 콜백).

**(B) 핵심 설계 = 갱신값을 작은 leaf 컴포넌트로 분리.** `setState`는 그 state를 가진 컴포넌트를 통째로 재호출하므로, 큰 테이블이 가격 state를 들면 틱마다 테이블 전체가 리렌더 대상(낭비). 갱신되는 값만 자기 자신의 잎 컴포넌트로 떼면 **그 컴포넌트만** 리렌더된다.
```tsx
function LivePrice({symbol}: {symbol: string}) {
    const [price, setPrice] = useState<number | null>(null);
    useEffect(() => {
        const ws = new WebSocket(`wss://example.com/quotes/${symbol}`);
        ws.onmessage = (e) => setPrice(JSON.parse(e.data).price);  // 틱마다 이 컴포넌트만 리렌더
        return () => ws.close();                                   // ⭐ cleanup
    }, [symbol]);
    return <span>{price === null ? "—" : price.toLocaleString()}</span>;
}
// 사용: <td><LivePrice symbol="005930" /></td>  — 테이블 구조·종목명은 리렌더 안 됨
```

**깜빡임 없는 이유:** 클라이언트 라우팅과 같은 뿌리. 새로고침 없이 VDOM diff로 **바뀐 텍스트 노드만 제자리 갱신**(`<span>`을 새로 만드는 게 아니라 안의 글자만 교체). DOM 노드가 사라졌다 생기지 않아 깜빡임 없음.

**성능(틱이 아주 잦을 때):**
- **throttle/배치:** 들어오는 틱을 모아 `requestAnimationFrame`(≈60fps)에 맞춰 한 번씩만 `setPrice`(사람 눈은 그 이상 못 봄).
- **ref로 React 우회(고급):** 가장 뜨거운 경로는 `useRef` + DOM 직접 쓰기로 리렌더 사이클 자체를 건너뜀.
  ```tsx
  const ref = useRef<HTMLSpanElement>(null);
  ws.onmessage = (e) => { ref.current!.textContent = JSON.parse(e.data).price; };  // 리렌더 0번
  return <span ref={ref} />;
  ```
  state를 안 바꾸니 리렌더 0, DOM 텍스트만 직접 갱신 → 초고빈도 최고 성능. 단 "React 바깥에서 DOM 조작"하는 예외 패턴이라 보통은 leaf 컴포넌트 방식으로 충분.

요약: **푸시 연결로 받아 → setter 통보 → 그 텍스트 노드만 갱신**, 설계 핵심은 "갱신값을 작은 leaf 컴포넌트로 격리". 깜빡임 없음 = 노드 재생성 없이 글자만 교체. 초고빈도면 rAF 배치, 극단적이면 `useRef` 우회.

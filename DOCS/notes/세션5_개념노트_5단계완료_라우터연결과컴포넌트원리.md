## 1. App.tsx의 역할 — "교통정리 본부"

지금까지 만든 것들을 트리로 보면:

```
App           ← 이번에 작업. URL ↔ 화면을 짝지어주는 최상단(라우팅)
 └ PostListPage   ← 화면(페이지). 데이터를 직접 import해 조립
    └ PostList  posts={...}   ← 카드 묶음(부품)
       └ PostCard post={...}  ← 카드 1장(부품) × 3
```

- **App = 앱의 최상단 "교통정리 컴포넌트".** "어떤 주소(URL)로 들어오면 어떤 화면을 그릴지"를 정하는 곳.
- 화면 내용을 직접 그리지 않고, **주소↔화면 매칭 규칙만** 담는다.
- 지금은 규칙이 1개(`/` → PostListPage)지만, 곧 `/post/:id`, `/login` 등으로 늘어난다.

### 주소창에서 화면까지 한 번에 보면

처음엔 `App.tsx`만 따로 보면 감이 끊기기 쉽다. 실제 흐름은 이렇게 이어진다:

```
main.tsx
 └ BrowserRouter
    └ App
       └ Routes
          └ Route path="/" element={<PostListPage />}
             └ PostListPage
```

- **`main.tsx`**: 앱 시작점. `<App />`를 실제 화면(root)에 꽂는다.
- **`BrowserRouter`**: 주소창(URL) 변화를 감시하는 바깥 래퍼. 라우터 기능이 동작할 "환경"을 깔아준다.
- **`App`**: 주소와 화면을 연결하는 규칙표를 적는다.
- **`Routes`**: 현재 URL을 보고 여러 `<Route>` 중 맞는 것을 고른다.
- **`Route`**: `path`가 조건, `element`가 "조건이 맞을 때 보여줄 화면"이다.

즉 `App.tsx`는 혼자 동작하는 파일이 아니라, **`BrowserRouter` 안에서 현재 주소를 받아와 화면을 고르는 중간 관리자**라고 보면 훨씬 자연스럽다.

## 2. 외부 라이브러리 import — `react-router-dom`은 왜 `src/`에 안 보이나

세션 중 "`react-router-dom` 파일이 안 보이는데 아직 안 만든 거 아니냐"는 의문이 있었음. **아니다 — 그건 우리가 만드는 파일이 아니라 외부 라이브러리(npm 패키지)다.**

import 대상은 두 종류로 나뉘고, **경로 모양으로 구분**한다:

| 종류 | 사는 곳 | import 경로 모양 | 예 |
|---|---|---|---|
| **내가 만든 파일** | `src/` 안 | **점(`.`)으로 시작**하는 상대경로 | `"./pages/PostListPage"`, `"../components/PostList"` |
| **외부 라이브러리** | `node_modules/` | **패키지 이름 그대로**(점 없음) | `"react"`, `"react-dom"`, `"react-router-dom"` |

- 그래서 `src/`를 아무리 뒤져도 `react-router-dom` 파일은 없다. `node_modules/`에 설치돼 있고, `package.json`의 `dependencies`에 `"react-router-dom": "^7.17.0"`으로 적혀 있다.
- **판별 요령:** import 경로가 점(`.`)으로 시작하면 "내 파일", 아니면 "외부 라이브러리".
- 증거: 이미 `main.tsx`에서 `import { BrowserRouter } from "react-router-dom"`을 쓰고 있다. 같은 라이브러리에서 이번엔 `Routes`, `Route`를 꺼내온 것.

## 3. named import vs default import — 중괄호의 유무

App.tsx에서 import 두 줄의 모양이 다른 이유. 내보내는 방식이 두 가지라서 가져오는 모양도 갈린다.

### default export (대표 1개)
```tsx
export default PostListPage;            // 내보내기: 파일당 딱 하나
import PostListPage from "./pages/..."; // 가져오기: 중괄호 없음, 이름 자유
```
"이 파일의 대표 선수". 한 파일에 **하나만** 가능. 가져올 때 이름을 마음대로 붙여도 됨.

### named export (이름표 붙은 여럿)
```tsx
export function Routes() {...}           // 라이브러리 쪽이 이름 달아 내보냄(여러 개)
import { Routes, Route } from "react-router-dom"; // 가져오기: 중괄호로 정확한 이름 콕 집음
```
한 파일에 여러 개 가능. 가져올 때 **정확한 이름을 중괄호로** 집어야 함.

### 한 줄 규칙
> **중괄호 `{ }` 있음 = named(이름 집어옴) / 없음 = default(대표 하나).**

그래서:
- `import { Routes, Route } from "react-router-dom"` → 라이브러리가 여러 부품을 named로 내보냄 → 중괄호
- `import PostListPage from "./pages/PostListPage"` → 우리가 default로 내보냄 → 중괄호 없음

## 4. `<>`로 감싸면 컴포넌트가 엘리먼트가 되는 원리 ⭐

> 핵심: **컴포넌트는 "함수(레시피)", 엘리먼트는 "그 레시피로 만든 주문서(명령서)".**

| 표기 | 정체 | 의미 |
|---|---|---|
| `PostListPage` | 함수 그 자체 | "화면 만드는 법(레시피)"이 적힌 함수. 아직 아무것도 안 만듦 |
| `<PostListPage />` | 엘리먼트 | "이 레시피로 화면 하나 만들어줘"라는 **주문서** |

### 꺾쇠(`< />`)가 하는 일 = "렌더 주문서"로 바꾸기

JSX는 빌드 시 평범한 JS 호출로 바뀐다. 도구 버전에 따라 `React.createElement(...)`일 수도 있고 `jsx(...)` 비슷한 호출일 수도 있지만, **핵심 뜻은 같다**:
```
<PostListPage />   →   React.createElement(PostListPage, null)
```
즉 꺾쇠 `< />`는 **"이 함수를 React가 이해하는 엘리먼트 형태로 바꿔라"**는 문법 설탕(syntactic sugar)이다. 결과물은 개념적으로 이런 자바스크립트 객체에 가깝다:
```js
{ type: PostListPage, props: {} }   // "PostListPage를 그려달라"는 메모
```
React는 나중에 이 객체를 보고 "아, PostListPage 함수를 실제로 호출해 화면을 그려야겠군" 하고 **자기 타이밍에** 실행한다.

중요한 건 **`<PostListPage />`가 HTML 문자열이 아니라 "렌더 지시서" 같은 객체**라는 점이다. 그래서 JSX를 쓰면 "지금 화면이 바로 그려진다"기보다, 먼저 **무엇을 그릴지 적은 주문서**가 만들어진다고 이해하면 된다.

### 그래서 element엔 `<PostListPage />`를 넣는다
- `element={PostListPage}` (함수 자체) ❌ → React는 함수가 아니라 "주문서(엘리먼트)"를 받아야 그림.
- `element={<PostListPage />}` (엘리먼트) ✅

여기서 `element`라는 prop 이름도 그대로 읽으면 쉽다:
- `path` = "어떤 주소일 때?"
- `element` = "그때 무엇을 보여줄까?"

즉 `<Route path="/" element={<PostListPage />} />`는 **조건 + 결과**를 한 줄로 적은 것이라고 보면 된다.

### 헷갈리지 말 것: `<>...</>` (빈 꺾쇠 = Fragment)
질문의 `<PostListPage />`와 **다른 문법**이다. `<>...</>`는 **Fragment** — 여러 엘리먼트를 묶을 때 쓰는 "투명 포장지"(실제 DOM 태그를 추가하지 않고 묶기만 함). 이름이 비슷해 헷갈리니 구분해 둘 것.

## 5. `path="/"` — 왜 `/`이고, "어차피 여기인데" 왜 쓰나 ⭐

### `/`는 "사이트의 맨 첫 주소(루트, 최상위)"

URL에서 도메인 뒤에 붙는 **경로(path) 부분**을 가리킨다:

| 전체 URL | path |
|---|---|
| `http://localhost:5173/` | `/` (아무것도 안 붙은 기본) |
| `http://localhost:5173/login` | `/login` |
| `http://localhost:5173/post/3` | `/post/3` |

### 오해 교정: `path="/"`는 "현재 위치"가 아니라 "조건(규칙)"

`<Route path="/" element={<PostListPage />} />`는 이렇게 읽어야 한다:

> "**만약** URL이 `/`이면 → PostListPage를 보여줘라."

`<Routes>`는 사용자가 주소창에 뭘 치든 **그때그때 현재 URL을 보고, 여러 Route 중 조건이 맞는 하나를 골라** 화면을 갈아끼운다. 지금은 Route가 1개라 "당연히 여기"처럼 느껴질 뿐, 본질은 **조건문**이다.

곧 이렇게 늘어난다:
```tsx
<Routes>
  <Route path="/"         element={<PostListPage />} />
  <Route path="/post/:id" element={<PostDetailPage />} />
  <Route path="/login"    element={<LoginPage />} />
</Routes>
```
이때 `path="/"`가 있어야 "루트로 오면 목록"이라는 규칙이 성립한다. **안 쓰면 루트 주소에서 아무 화면도 안 뜬다.**

> 한 줄: `path="/"`는 "지금 위치 선언"이 아니라 **"어느 주소에 어느 화면을 매달지 정의하는 라벨(조건)"**.

## 6. `export default PostListPage` — 왜 괄호를 떼야 하나

세션 중 `export default PostListPage()` (괄호 있음)로 썼다가 버그였음.

| 코드 | 의미 | 결과 |
|---|---|---|
| `export default PostListPage;` | 이 **함수(부품) 자체**를 내보냄 | ✅ React가 필요할 때 호출해 그림 |
| `export default PostListPage();` | 이 함수를 **지금 당장 실행**해 그 **리턴값(JSX 엘리먼트)** 을 내보냄 | ❌ 컴포넌트가 아니라 결과물을 내보냄 |

- 컴포넌트는 "React가 자기 타이밍에 호출"해야 하는데(§4), 괄호를 붙이면 내가 즉시 실행해버려서 React가 컴포넌트로 못 쓴다.
- 규칙: **내보낼 땐 함수 "이름만"** (`export default App;`, `export default PostListPage;`). 괄호 금지.
- 실제로 import하는 쪽에서는 `PostListPage`를 **컴포넌트 함수라고 기대**하고 `<PostListPage />`처럼 쓰는데, 괄호를 붙여 export하면 이미 "실행 결과물"이 들어 있어서 역할이 어긋난다.

## 7. App도 컴포넌트인가? — 그렇다

App은 React 컴포넌트의 조건을 전부 만족한다:

1. **함수다** → `function App() {...}`
2. **대문자로 시작** → `App` (React는 대문자로 시작해야 컴포넌트로 인식. 소문자면 HTML 태그로 취급)
3. **JSX를 리턴한다** → `return (<Routes>...</Routes>)`

그리고 실제로 main.tsx에서 **엘리먼트로 호출**된다:
```tsx
<BrowserRouter>
  <App />     {/* App을 엘리먼트로 써서 화면에 그림 */}
</BrowserRouter>
```

→ App은 PostListPage·PostList·PostCard와 **똑같은 컴포넌트**이고, 역할만 다르다. App은 "최상단 교통정리 컴포넌트", 나머지는 그 아래 화면·부품. React 앱은 이렇게 컴포넌트가 컴포넌트를 품는 **트리 구조**다(§1 트리 참고).

## 8. `/>` 만 쓰는 문법 — 셀프클로징(self-closing)

`/>`로 끝내는 건 **"이 태그는 안에 자식(내용)이 없다"**는 뜻이다.

### 태그를 닫는 두 가지 방식

**① 여는 태그 + 닫는 태그 (자식이 있을 때)**
```tsx
<h1>변명 게시판</h1>          // 사이에 텍스트
<div>
  <PostList />              // 사이에 다른 엘리먼트
</div>
```

**② 셀프클로징 `/>` (자식이 없을 때)**
```tsx
<PostList posts={MOCK_POSTS} />
<Route path="/" element={<PostListPage />} />
```

### 원리
`<PostList />`는 `<PostList>`(열기) + `</PostList>`(닫기)를 **한 방에 합친 것**. 끝의 `/`가 "여기서 바로 닫는다"는 표시.
```tsx
<PostList>   // ❌ 닫는 짝이 없어서 에러
<PostList /> // ✅ 자식 없이 바로 닫음
```

### 핵심 규칙: JSX는 모든 태그가 반드시 닫혀야 한다
HTML은 `<img src="...">`처럼 `/` 없이 써도 봐주지만, **JSX는 안 봐준다** → `<img ... />`로 꼭 `/`를 붙여야 한다.

우리 코드에 적용:
- `<Route ... />`, `<PostListPage />`, `<PostList ... />` → 자식 없음 → 셀프클로징
- `<div>...</div>` → 안에 h1·PostList 들어감 → 여닫기 둘 다

## 9. App은 왜 인자가 없나 — 인자 = props 받는 자리

컴포넌트 함수의 **인자 자리 = props(부모가 넘겨주는 데이터)를 받는 자리**다. App에 인자가 없는 건 **아무도 App한테 props를 안 넘기기 때문**.

```tsx
function PostList({ posts }) {...}   // 부모가 posts를 넘기니 받을 인자 있음
function App() {...}                 // 넘겨받는 게 없으니 빈 괄호
```

main.tsx에서 App을 쓰는 모습:
```tsx
<App />     // 아무 속성(props)도 안 넘김 → 받을 인자도 없음
```

- 부모가 `<컴포넌트 속성={값} />`로 뭔가 넘기면 → 자식 함수는 인자(`{ 속성 }`)로 받음.
- 아무것도 안 넘기면 → 빈 괄호 `()`.
- App은 앱의 **최상단**이라 위에서 내려줄 부모 데이터가 없다. (인자를 써도 되긴 함 — `function App(props)` — 안 쓸 거면 빈 괄호가 깔끔.)

여기서 중요한 건 **"props가 없으면 덜 중요한 컴포넌트"가 아니라는 점**이다. props 유무는 단지 **그 컴포넌트가 바깥에서 데이터를 받는 역할인지 아닌지**를 말해줄 뿐이다.

## 10. 왜 어떤 컴포넌트는 받는 데이터(props)가 없나 ⭐

핵심은 **"데이터를 어디서 구하느냐"가 컴포넌트마다 다르기 때문.** 데이터를 얻는 경로가 두 가지다.

### 경로 A — 부모한테서 받는다 (props 있음)
```tsx
function PostCard({ post }) {...}   // 부모가 준 post로 그림
function PostList({ posts }) {...}  // 부모가 준 posts로 그림
```
"**받은 것만 그리는 부품**". 스스로 데이터를 어디서 가져올지 모른다. 누가 주면 그걸로 그릴 뿐(= 표현 전담 부품). 그래서 재사용이 쉽다.

### 경로 B — 스스로 데이터를 구해온다 (props 불필요)
```tsx
function PostListPage() {
  // 부모가 안 줘도, 내가 직접 import해서 데이터를 구함
  return <PostList posts={MOCK_POSTS} />;
}
```
PostListPage는 props를 안 받는다. 대신 **자기가 직접 `MOCK_POSTS`를 import**해 데이터를 마련하고, 자식(PostList)에게 내려준다. (다음 단계에선 import 대신 `fetchPosts()` API 호출이나 `useState`로 직접 데이터를 만든다 — 6단계.)

### 트리로 본 데이터 흐름 (단방향)
```
App           ← props 없음. 역할이 "라우팅"뿐이라 데이터 자체가 필요 없음
 └ PostListPage   ← props 없음. 데이터를 "스스로 구해서(import)" 아래로 내려줌  ★데이터 출발점
    └ PostList  posts={...}   ← 받음. 받은 걸 .map으로 펼침
       └ PostCard post={...}  ← 받음. 받은 한 장을 그림
```

데이터는 **위에서 한 번 출발(PostListPage)** 해서 **아래로 흘러내려간다.** 그래서:
- **상단/구조 컴포넌트**(App, PostListPage): 받을 부모가 없거나 스스로 구하니 props가 적거나 없음.
- **아래쪽 부품**(PostList, PostCard): 위에서 내려준 걸 받아야 하니 props 있음.

### props가 없는 두 가지 경우 정리
1. **데이터가 아예 필요 없다** — App처럼 라우팅·레이아웃 같은 "구조" 역할만.
2. **데이터를 스스로 마련한다** — PostListPage처럼 import/API/useState로 직접 구해옴(그래서 "페이지" 컴포넌트들이 보통 이렇다).

> React 핵심 원리: **데이터는 위→아래 단방향으로 흐른다(one-way data flow).** 위에서 구하거나 만들어, 아래 부품들에 props로 뿌린다.

초심자 눈에는 "왜 어떤 애는 props가 있고 어떤 애는 없지?"가 불규칙하게 느껴질 수 있는데, 실제 기준은 꽤 단순하다:
- **화면 조립/데이터 출발점**에 가까울수록 props가 없을 수 있다.
- **작은 표현 부품**일수록 props를 받는 경우가 많다.

이 감각이 잡히면 나중에 `useState`, `useEffect`, Context를 배울 때도 "어디서 상태를 만들고 어디로 내려보낼지"가 훨씬 덜 헷갈린다.

## 11. 오늘 핵심 요약 (6줄)

1. App은 **URL↔화면을 짝짓는 최상단 컴포넌트**. `<Routes>` 안에 `<Route path="/" element={<PostListPage />} />`로 규칙을 적는다.
2. 전체 흐름은 `main.tsx → BrowserRouter → App → Routes → Route → PostListPage`. BrowserRouter가 주소 변화를 감시하고, App이 그 규칙표를 들고 있다.
3. `react-router-dom`처럼 **점(`.`) 없이 import하면 외부 라이브러리**(`node_modules`), 점으로 시작하면 내 파일. named는 중괄호, default는 중괄호 없음.
4. `<PostListPage />`(꺾쇠)는 함수 자체가 아니라 **React에게 넘기는 엘리먼트(주문서)**. 그래서 element엔 `<... />`를 넣고, export엔 괄호 없이 함수 이름만 쓴다.
5. `path="/"`는 "현재 위치"가 아니라 **"이 주소면 이 화면"이라는 조건(규칙)**. `/`는 사이트 루트 주소이고, `element`는 그 조건에 맞을 때 보여줄 화면이다.
6. props 유무는 **데이터를 위에서 받느냐(PostList/PostCard) vs 스스로 구하거나 필요 없느냐(PostListPage/App)**의 차이 — 데이터는 단방향으로 위→아래로 흐른다.

## 1. TypeScript — TypeScript는 무엇이며, JavaScript와 무엇이 다른가? 왜 쓰는가?


### 내 답변 1

TypeScript는 Type 검사를 지원하는 자바스크립트이다. JS는 동적 언어라 실행하기 전까지 숨은 오류가 잘 드러나지 않는다. 
때문에 크고 중요한 시스템일수록 실행 전에 오류 방지와 개발 편의를 위해 TS를 사용한다.
TS에서 말하는 type이란 흔히 생각하는 boolen, number 등의 type이 아니다. 이것들은 primative type. 즉, 원형적인 타입이라는 이름으로 따로 불리고 있다.
TS의 type은 데이터의 모양을 말한다. 어떠한 데이터가 와야 하는지, 키는 무엇이고 primative type은 무엇인지를 미리 정해놓는다. 특히 여러 필드가 있는 객체의
모양을 정의할 때 효과적으로 작용한다.

## 2. React — React는 무엇이고, 어떤 문제를 해결하려고 등장했는가?

### 내 답변 2

React는 웹이 앱으로 진화하면서 등장했다. 웹이 앱으로 진화하면서 수 많은 페이지, 기능, 요소들을 갖게되었다. 그래서 기존의 HTML 문법만으로는 이것들을 효율적으로 다룰 수 없었다.
그래서 Solo Page App이면서 상태로 element들을 관리하는 React가 등장했다. 
React는 화면을 여러 요소들로 나눈다. 그렇게 하면 요소를 재사용할 수 있다. 또한 화면이 바뀌었을 때 전체를 재랜더링 하는 것이 아니라 해당 요소만 재랜더링한다. 왜냐하면 모든 요소들은
자기만의 '상태'를 갖기 때문이다. 상태에 변화가 있을 시 setstate 함수가 실행되고 화면이 바뀌었음을 알린다. 바뀐 요소가 JSX 형태로 브라우저에게 전달되고 클라이언트 측에서는 해당 요소만
패치한다. 때문에 전체 새로고침이 일어나는 게 아니라 변경된 부분만 비동기적으로 업데이트된다. 이를 Server Side Rendering이라고 한다.
주목할 것은 React는 이미 여러 요소들이 합쳐진 HTML를 보내는 것이 아니라 "여기에는 이게 들어갈 예정이야." 식의 명령만 브라우저에게 전달한다는 것이다. 브라우저는 이를 받아 hydrating 하고
그제서야 실제 작동하는 HTML로 만들어진다. 이렇듯 서버에서 화면을 만들어서 보내는 것이 아니기 때문에 맨 처음 전체화면 렌더링 시 브라우저가 모든 것을 그리느라 화면 로딩이 다소 느릴 수 있다는 단점이 존재한다.

## 3. CSR (Client Side Rendering) — CSR이란 무엇이고, 화면이 어떻게 그려지는가?

바로 위에서 설명하였다. 간단하게 서버가 아닌 클라이언트 측에서 필요한 데이터와 요소들을 직접 찾아 화면을 만드는 것을 말한다. URL이 바뀌고 다른 페이지로 이동할 때
전체 페이지를 바꾸는 것이 아닌 새로 바뀐 요소만 업데이트한다. 중요한건 이 업데이트 또한 서버가 하는 것이 아니라는 것이다. 브라우저가 이전에 받아뒀던 데이터와 요소들 중 해당 URL과 변경된 상태를
고려하여 알아서 화면을 렌더링한다.

## 4. SSR (Server Side Rendering) — SSR이란 무엇이고, CSR과 비교한 장단점은?

SSR은 서버에서 클라이언트가 URL로 요청한 데이터를 바로 쓸 수 있게 모아서 보내주는 것이다. 첫 렌더링이 빠르다는 장점이 있다.

## 5. 컴포넌트 설계 — 컴포넌트를 잘 설계한다는 건 무엇이고, 어떤 기준으로 나누는가?


적은 양의 컴포넌트를 재사용하며 부모로부터 받은 props를 통해 다채로운 화면을 효율적으로 그리는 설계가 좋은 설계이다. 컴포넌트를 나눌 때는 화면에 보이는 요소들을 추상화하여 나타낸다.
예를 들면 카드, 검색창, 상세 페이지 등이 있다.

## 6. 상태(State) / Props — State와 Props는 각각 무엇이고, 둘의 차이는?

상태는 각 엘리먼트들이 가지고 있는 현재 모습이다. 상태는 새로운 데이터나 계산에 의해 실시간으로 변화한다. 사용자마다 다른 상태를 볼 수도 있다. Props는 부모가 주는 재료이다.
부모로부터 재료를 받아 엘리먼트들을 만들기 때문에 각기 다른 데이터를 담을 수 있다. 말하자면 상태와 Props를 통해 빈 껍데기인 엘리먼트가 실제 고유한 데이터를 담게 된다.

## 7. 이벤트(Event) — React에서 이벤트란 무엇이고, 어떻게 다루는가?

이벤트는 사용자가 어떤 행위를 했을 때 발생하고 발생하면 미리 만들어진 이벤트 핸들러가 작동해 자동으로 특정 요소의 상태를 변경한다. 

## 8. 라이프사이클(Lifecycle) — 컴포넌트 라이프사이클이란 무엇이고, 대표 단계는?

컴포넌트가 만들어지고 사라지기까지의 전 과정을 말한다. 첫 렌더링 된 뒤 상태가 변경되면 패치가 일어난다. 그러다가 사용자의 조작에 따라 현재 화면에서 더 이상 필요없게 된 요소들은 사라진다.

## 9. 라우팅(Routing) — 라우팅이란 무엇이고, 프론트엔드에서 왜 필요한가?

라우팅은 URL로 화면을 전환하는 것을 말한다. api를 관리하는 App에서 이루어진다. 제품 상세페이지 등 상태로 관리하는 것 보다 URL로 사용자에게 노출해 주소 복사 등을 가능하게 해야 하는 요소들에게 필요하다.

## 10. 상태관리(State Management) — 상태관리란 무엇이고, 왜 별도로 필요해지는가?

상태관리는 어떤 엘리먼트가 상태를 가지고 있을지를 관리하는 것이다. 같은 상태가 여러 엘리먼트에 있게 되면 중복 업데이트가 일어날 수 있다. 때문에 그 경우 가장 가까운 둘의 공통 부모가 상태를 관리한다.
또 상태 또한 실제 계산이 필요한 부분에만 적용한다.

## 11. 테스트(Test) — 프론트엔드에서 테스트란 무엇이고, 왜 하는가?

프론트엔드 테스트는 단위 테스트 스모크 EtE로 나눌 수 있다. 실제 화면이 의도한 대로 나타나는지 검사하는 것이 중요하다. 때문에 단순 기능을 평가하는 단위 테스트부터 실제 사용자가 보는 화면을
테스트하는 엔드 투 엔드 테스트까지 많은 테스트 단계를 거친다.

## 12. UI 프레임워크 — UI 프레임워크란 무엇이고, 어떤 점이 편한가? (예시도)

UI에서 자주 반복되어 사용되는 것들을 미리 만들어두고 간단하게 가져다 쓸 수 있도록 한다. 일관적인 UI를 만들 수 있고 짧고 추상화되어 가독성이 높은 코드를 쓸 수 있다.
